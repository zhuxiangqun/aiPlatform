# aiPlat-core 架构师指南

> 本文档为架构师提供 aiPlat-core 核心层的架构设计指导，帮助架构师理解核心层的设计理念、技术选型和模块边界。

---

## 概述

aiPlat-core 是 AI 中台系统的核心层，位于四层架构的第二层（Layer 1）。本层不关心如何暴露（HTTP/CLI）也不关心如何存储（数据库/文件），专注于封装 AI 核心能力和业务逻辑。

**核心定位**：

- 智能体的生命周期管理
- 技能的注册与执行
- 工作流的编排与调度
- 记忆系统的管理
- 知识库的构建与检索
- 工具系统的集成

**依赖关系**：

- 向上：依赖 aiPlat-platform（通过 CoreFacade）
- 向下：依赖 aiPlat-infra（通过工厂接口）
- 不依赖：app 层

---

## 整体架构

### 分层结构

aiPlat-core 采用分层模块化架构：

| 层次 | 模块 | 说明 |
|------|------|------|
| 框架层 | harness | 智能体框架基础设施 |
| 编排层 | orchestration | 工作流编排能力 |
| 实现层 | agents, skills | 智能体和技能的具体实现 |
| 支撑层 | memory, knowledge, tools | 记忆、知识、工具的支撑能力 |
| 服务层 | services, models | 公共服务和模型封装 |

### 模块依赖关系

模块之间的依赖遵循自下而上的原则：

```
harness (无依赖，基础层)
    ↑
orchestration ← harness
agents ← harness, memory, knowledge, tools, models
skills ← harness, tools, models
memory ← services
knowledge ← services, models
tools ← services
services ← aiPlat-infra
models ← aiPlat-infra
```

## 核心抽象概念

### 智能体（Agent）

智能体是核心层最重要的抽象，代表一个具备 AI 能力的执行单元。

**概念定义**：

| 维度 | 描述 |
|------|------|
| 身份 | 唯一标识，包含名称、类型、版本等元信息 |
| 能力 | 可执行的技能和工具集合 |
| 记忆 | 短期记忆（当前会话）+ 长期记忆（跨会话） |
| 状态 | 当前执行状态（初始化→就绪→运行→暂停→终止） |
| 目标 | 要完成的任务目标 |

**设计原则**：

- 智能体应该是无状态的，状态通过外部传入
- 智能体应该支持并行执行多个任务
- 智能体应该支持中断与恢复

### 技能（Skill）

技能是智能体可执行的能力单元，是完成任务的具体手段。

**概念定义**：

| 维度 | 描述 |
|------|------|
| 名称 | 唯一标识 |
| 描述 | 功能说明 |
| 输入 | 输入参数定义 |
| 输出 | 输出结果定义 |
| 策略 | 执行策略（同步/异步/重试） |

**设计原则**：

- 技能应该是无副作用的纯函数
- 技能应该有清晰的输入输出定义
- 技能应该支持幂等执行
- 技能应该支持超时控制

### 记忆（Memory）

记忆是智能体的信息存储系统。

**概念定义**：

| 类型 | 用途 | 特征 |
|------|------|------|
| 短期记忆 | 当前会话信息 | 快速访问，时效性 |
| 长期记忆 | 跨会话经验 | 持久可靠，可检索 |
| 工作记忆 | 任务执行状态 | 临时缓存 |

**设计原则**：

- 短期记忆应该快速访问，支持高频读写
- 长期记忆应该持久可靠，支持语义检索
- 记忆应该支持自动清理，防止无限增长

### 工作流（Workflow）

工作流定义了任务的执行步骤和条件。

**概念定义**：

| 元素 | 描述 |
|------|------|
| 步骤 | 任务执行的原子单元 |
| 条件 | 根据执行结果决定下一步 |
| 循环 | 重复执行某些步骤 |
| 并行 | 同时执行多个步骤 |

**设计原则**：

- 工作流定义应该是声明式的
- 工作流执行应该是可观测的
- 工作流应该支持版本管理
- 工作流应该支持热更新

## 技术选型

### 编程语言

- **Python 3.10+**：核心层使用 Python 开发，利用其丰富的 AI 生态

### 异步框架

- **asyncio**：Python 原生异步支持
- **aiohttp**：异步 HTTP 客户端
- **asyncpg**：异步 PostgreSQL 客户端

### 消息队列

- **Redis**：轻量级消息队列，用于任务分发
- **RabbitMQ**：企业级消息队列，用于可靠消息传递

### 向量数据库

- **Milvus**：高性能向量数据库，用于知识检索
- **Qdrant**：轻量级向量数据库，用于小规模场景

### 缓存

- **Redis**：内存缓存，用于热点数据缓存

## 模块职责边界

### harness 模块

**职责**：提供智能体框架基础设施

**边界**：

- 负责：Agent 的生命周期、心跳监控、健康分数、状态追踪
- 不负责：具体的业务逻辑、技能执行、工作流编排

### orchestration 模块

**职责**：提供工作流编排能力

**边界**：

- 负责：工作流定义、执行调度、条件分支、并行控制
- 不负责：具体的 Agent 实现、技能执行

### agents 模块

**职责**：提供各类智能体实现

**边界**：

- 负责：Agent 类型的定义、执行逻辑、配置管理
- 不负责：底层框架、心跳监控等基础设施

### skills 模块

**职责**：提供技能系统

**边界**：

- 负责：Skill 的定义、注册、执行
- 不负责：具体的工具实现、知识存储

### memory 模块

**职责**：提供记忆系统

**边界**：

- 负责：短期/长期记忆的存取、检索、清理
- 不负责：具体的存储实现（由 infra 层提供）

### knowledge 模块

**职责**：提供知识管理能力

**边界**：

- 负责：知识库管理、文档处理、检索排序
- 不负责：向量存储实现（由 infra 层提供）

### tools 模块

**职责**：提供工具系统

**边界**：

- 负责：工具的定义、注册、调用、权限控制
- 不负责：具体工具的实现

## 关键架构机制

### 自我进化机制

> 详细设计见 [Harness 自我进化机制](../../harness/index.md#5-自我进化机制)

**解决的问题**：

- Agent 需要根据执行历史自动优化策略
- 静态配置无法适应动态环境
- 性能调优需要人工介入

**设计要点**：

- 性能驱动：基于实际执行数据进行优化
- 分层进化：确保优化策略的安全性和有效性（LOCAL → PUSH → PROD）
- 可追溯：所有进化决策可回溯和回滚

### 三层反馈循环

> 详细设计见 [Harness 三层反馈循环](../../harness/index.md#6-三层反馈循环系统)

**解决的问题**：

- 优化策略的安全性问题
- 如何在不影响生产的前提下验证优化
- 异常时的快速回滚

**设计要点**：

- 每层独立存储，互不干扰
- 上升条件：性能提升超过阈值
- 下降条件：错误率超过阈值

### 门面模式（CoreFacade）

**解决的问题**：

- platform 直接依赖 core 内部模块会导致耦合紧密
- core 重构会影响 platform
- 模块 API 变化需要同步修改

**架构设计**：

- 统一入口：platform 只需依赖 CoreFacade
- 隔离变化：core 内部重构不影响 platform
- 版本演进：facade 接口独立版本控制

## 模块间依赖规则

| 模块 | 可依赖 | 禁止依赖 |
|------|--------|----------|
| harness | 无（基础层） | - |
| orchestration | harness, agents, skills, tools | services, memory, knowledge |
| agents | harness, memory, knowledge, tools, models | services |
| skills | harness, tools, models | memory, knowledge |
| memory | services | agents, skills, tools |
| knowledge | services, models | agents, skills |
| tools | services | agents, skills |

## 与 infra 层边界

**core 层从 infra 获取什么**：

- **计算资源抽象**：通过工厂接口获取，不关心具体实现
- **数据存储抽象**：通过数据访问接口获取，不关心具体技术
- **模型服务抽象**：通过模型客户端获取，不关心提供商

**core 层不关心什么**：

- 资源调度细节（CPU/GPU/内存分配）
- 数据持久化策略（备份、恢复、副本）
- 服务发现机制（域名解析、负载均衡）

## 性能优化

### 异步执行

- 使用 asyncio 实现异步执行
- 使用连接池管理数据库连接
- 使用缓存减少重复计算

### 并行处理

- 使用并行执行提高吞吐量
- 使用批处理减少 API 调用
- 使用流式处理减少内存占用

### 缓存策略

- 使用 Redis 缓存热点数据
- 使用本地缓存减少网络开销
- 使用缓存预热提高响应速度

## 监控与告警

### 监控指标

- 智能体执行次数和耗时
- 技能调用次数和成功率
- 模型调用次数和成本
- 记忆存储大小和检索延迟

### 告警规则

- 智能体执行失败率超过阈值
- 模型调用延迟超过阈值
- 记忆存储空间不足
- 系统资源使用率过高
