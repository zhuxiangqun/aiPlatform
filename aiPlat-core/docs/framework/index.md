# 框架基础

> aiPlat-core 基于 LangGraph 和 LangChain 构建，本文档描述框架集成设计和核心组件的详细设计。

---

## 一、框架概述

aiPlat-core 的技术栈建立在 LangGraph 和 LangChain 之上，利用这两个框架的能力构建 AI 中台的核心能力。

| 框架 | 定位 | aiPlat-core 中的角色 |
|------|------|---------------------|
| **LangGraph** | 基于图结构的编排框架 | 工作流编排、智能体协作 |
| **LangChain** | LLM 应用开发工具链 | 模型调用、工具链、记忆管理 |

---

## 二、整体层次架构

### 2.1 层次结构图

```
┌─────────────────────────────────────────────────────────────────────┐
│                         apps/ (应用层)                              │
│              (Agents, Skills, Tools 具体实现)                       │
├─────────────────────────────────────────────────────────────────────┤
│                       harness/ (操作系统层)                          │
│    (生命周期管理、心跳监控、状态追踪、自我进化、反馈循环)              │
├─────────────────────────────────────────────────────────────────────┤
│                       adapters/ (适配器层)                          │
│                    (LLM 适配器、外部集成)                            │
├─────────────────────────────────────────────────────────────────────┤
│                      LangGraph (图结构编排)                        │
│                 (Workflow, State, Nodes, Edges)                      │
├─────────────────────────────────────────────────────────────────────┤
│                      LangChain (基础工具链)                         │
│                   (Models, Memory, Tools, Prompts)                  │
├─────────────────────────────────────────────────────────────────────┤
│                     aiPlat-infra (基础设施层)                       │
│              (LLM Clients, Vector Stores, Databases)                │
└─────────────────────────────────────────────────────────────────────┘
```

### 2.2 各层职责

| 层级 | 定位 | 核心功能 |
|------|------|----------|
| **apps/** | 应用层 | Agent、Skill、Tool 具体实现 |
| **harness/** | 操作系统层 | 生命周期、协作协调、观测控制、自我进化 |
| **adapters/** | 适配器层 | 外部 LLM 服务集成 |
| **LangGraph** | 编排引擎 | 图结构编排、工作流定义 |
| **LangChain** | 工具链 | LLM 调用、工具接口、记忆组件 |
| **aiPlat-infra** | 基础设施 | 底层存储、计算资源 |

### 2.3 关系类比

```
LangChain     → 汽车的发动机、变速箱（基础动力系统）
LangGraph     → 汽车的方向盘、刹车系统（操控与协调）
Harness       → 汽车的仪表盘、监控系统、导航系统（全局控制与优化）
adapters/     → 汽车的油箱（能源供给）
```

### 2.4 依赖关系总结

| 层级 | 依赖方向 | 说明 |
|------|----------|------|
| **apps/** | 依赖 harness/ | 调用框架能力 |
| **harness/** | 依赖 adapters/ + LangChain | 使用模型和工具 |
| **adapters/** | 依赖 LangChain | 封装外部服务 |
| **LangChain** | 依赖 LangGraph | 构建于图结构 |
| **LangGraph** | 依赖 aiPlat-infra | 底层资源 |

## 三、Harness 操作系统层

> **重要**：Harness 是 aiPlat-core 的"操作系统"，为所有核心模块提供基础设施服务。

### 3.1 Harness 定位

Harness 不是普通的业务模块，而是整个 aiPlat-core 的基础架构层：

| 维度 | 描述 |
|------|------|
| **角色** | 智能体框架基础设施 |
| **职责** | 生命周期管理、心跳监控、状态追踪、自我进化、反馈循环 |
| **地位** | 底层基础层，所有其他模块依赖 Harness |

### 3.2 为什么是操作系统

Harness 被称为"操作系统"的原因：

| 功能分类 | 核心能力 | 说明 |
|---------|---------|------|
| **资源管理** | Agent 生命周期 | 管理 Agent 实例的创建、运行、停止、销毁 |
| | 上下文管理 | 维护会话上下文、状态传递、上下文切换 |
| | 提示词管理 | 模板注册、变量替换、版本控制 |
| | 记忆资源 | 短期/长期记忆的分配、检索、清理 |
| | 知识资源 | 知识库索引、检索优化、缓存管理 |
| **调度执行** | 任务调度 | 协调多个 Agent 的执行顺序和资源竞争 |
| | 工具调度 | 工具调用队列、优先级、并发控制 |
| | 技能调度 | 技能依赖解析、执行顺序、结果传递 |
| **状态管理** | 系统状态 | 维护整个系统的运行状态和配置 |
| | 执行状态 | 追踪执行链路、记录状态变更、支持恢复 |
| | 会话状态 | 管理会话上下文、多轮对话状态 |
| **故障恢复** | 错误检测 | Agent 故障检测、超时处理、异常捕获 |
| | 重试机制 | 自动重试、降级策略、故障转移 |
| | 状态恢复 | 从检查点恢复、状态回滚、断点续传 |
| **进化优化** | 性能追踪 | 记录执行指标、成功率、响应时间 |
| | 自动调优 | 根据运行时数据自动优化系统表现 |
| | 反馈循环 | LOCAL/PUSH/PROD 三层反馈机制 |

**与通用操作系统类比**：

| 操作系统概念 | Harness 对应 | 说明 |
|-------------|-------------|------|
| 进程 | Agent 实例 | 独立执行单元 |
| 线程 | Agent 执行任务 | 并发执行 |
| 调度器 | 编排引擎 | 任务调度 |
| 文件系统 | Memory/Knowledge | 数据存储 |
| 设备驱动 | Tools | 外部资源访问 |
| 系统监控 | 状态追踪/心跳 | 运行监控 |
| 环境变量 | AgentContext | 上下文传递 |
| 配置文件 | PromptTemplate | 提示词模板 |
| 进程间通信 | ContextService | 上下文共享 |

### 3.3 与 LangGraph/LangChain 的关系

```
Harness → adapters/ → LangChain → LangGraph → aiPlat-infra
         ↓
    (通过 LCEL 表达式调用模型)
```

## 四、LangGraph 集成

### 4.1 核心概念映射

LangGraph 的核心概念与 aiPlat-core 的映射关系：

| LangGraph 概念 | aiPlat-core 实现 | 说明 |
|----------------|-----------------|------|
| **Node** | Agent / Skill / Tool | 执行步骤的原子单元 |
| **Edge** | Workflow / Condition | 节点间的执行顺序和跳转 |
| **State** | AgentContext / WorkflowContext | 节点间传递的上下文 |
| **Cycle** | Loop / Multi-turn | 迭代执行和多轮对话 |

### 4.3 状态管理

**状态结构设计**：

| 字段 | 类型 | 说明 |
|------|------|------|
| messages | List[Message] | 对话历史 |
| context | Dict | 任务上下文 |
| tools_calls | List[ToolCall] | 待执行的工具调用 |
| results | Dict | 工具执行结果 |
| step_count | int | 已执行步骤数 |
| metadata | Dict | 元信息 |

**状态更新规则**：

- 每个 Node 可以读取和写入 State
- State 更新是原子性的，保证一致性
- 支持中间状态检查点，用于恢复

## 六、Agent 详细设计

### 6.1 Agent 类型与模式

| Agent 类型 | 模式 | 适用场景 | 详细说明 |
|------------|------|----------|----------|
| **ReAct Agent** | Reason + Act 循环 | 需要推理和工具调用的任务 | 见 [Agent 设计模式](./patterns.md) |
| **Plan-Execute Agent** | 先规划后执行 | 复杂任务，需要步骤分解 | 见 [Agent 设计模式](./patterns.md) |
| **Conversational Agent** | 对话模式 | 多轮交互，问答场景 | - |
| **Tool-Using Agent** | 工具为主 | 需要频繁调用外部工具 | 见 [Agent 设计模式](./patterns.md) |
| **RAG Agent** | 检索+生成 | 需要知识库支持的回答 | - |
| **Multi-Agent** | 协作模式 | 复杂任务，需要多角色 | 见 [Agent 设计模式](./patterns.md) |
| **TriAgent** | Planner-Generator-Evaluator | 需要评估验证的复杂任务 | 见 [Harness TriAgent](./harness/index.md) |
| **Reflection Agent** | 自我审视+修正 | 代码审查、内容润色 | 见 [Agent 设计模式](./patterns.md) |
| **Planning Agent** | 任务分解+执行 | 自动化流程、报告生成 | 见 [Agent 设计模式](./patterns.md) |

> **提示**：详细的 Agent 设计模式说明（含 Reflection 和 Planning），请参阅 [Agent 设计模式](./patterns.md)。

### 6.2 执行流程

Agent 的执行流程包含Skill 和 Tool 两个层次的决策：

```
┌─────────────────────────────────────────────────────────────┐
│                    Agent 执行流程                           │
└─────────────────────────────────────────────────────────────┘

1. 输入任务
   │
   ▼
2. LLM 推理决定下一步
   │
   ▼
3. 判断需要什么能力？
   │
   ├─► 需要内部能力 (Skill)
   │       │
   │       ▼
   │   执行 Skill (如文本生成、代码分析)
   │       │
   │       ▼
   │   更新状态 → 回到步骤 2
   │
   ├─► 需要外部资源 (Tool)
   │       │
   │       ▼
   │   执行 Tool (如 API调用、数据库查询)
   │       │
   │       ▼
   │   更新状态 → 回到步骤 2
   │
   └─► 无需执行
           │
           ▼
       生成最终答案 → 结束
```

**Skill vs Tool 的判断依据**：

| 判断条件 | 使用 Skill | 使用 Tool |
|---------|-----------|----------|
| **执行位置** | Agent 内部 | Agent 外部 |
| **能力类型** | 内置能力 | 外部接口 |
| **典型案例** | 文本生成、代码分析、数据转换 | API 调用、数据库查询、网络搜索 |
| **是否需要外部资源** | 否 | 是 |
| **执行方式** | 直接调用内部逻辑 | 通过接口调用外部服务 |

**执行示例**：

```
用户任务："帮我分析这段代码的性能瓶颈"

Agent 执行流程：
1. 输入任务：分析代码性能瓶颈
2. LLM 推理：需要先理解代码结构
3. 判断：需要内部能力 → 执行 CodeAnalysisSkill
4. Skill 返回：代码结构分析结果
5. 更新状态，回到步骤 2
6. LLM 推理：需要查询历史性能数据
7. 判断：需要外部资源 → 执行 DatabaseTool
8. Tool 返回：历史性能数据
9. 更新状态，回到步骤 2
10. LLM 推理：信息足够，可以回答
11. 生成最终答案：性能瓶颈分析报告
```

**Harness Engineering 最佳实践**：

参考 [Harness 文档](./harness/index.md) 了解以下核心原则：

| 概念 | 说明 |
|------|------|
| **Sprint Contract** | 冲刺合约，开发前明确功能范围和验收标准 |
| **File-based Communication** | 文件化通信，通过结构化文件进行 Agent 间协作 |
| **Value Decay Curve** | 价值衰减曲线，不同反馈类型价值衰减速率不同 |
| **Format Affinity** | 格式亲和性，模型对特定输出格式的偏好 |
| **TriAgent** | Planner-Generator-Evaluator 三智能体架构 |
| **Rippability** | 可拔插性，组件可随模型能力提升而移除 |

### 6.3 状态转换

| 状态 | 说明 | 下一状态 |
|------|------|----------|
| INIT | 初始状态 | REASONING |
| REASONING | 模型推理中 | ACTING / FINISHED |
| ACTING | 执行工具中 | REASONING |
| FINISHED | 执行完成 | - |
| ERROR | 执行错误 | REASONING / FINISHED |

## 七、Skill 详细设计

### 7.1 Skill 结构

| 组件 | 说明 |
|------|------|
| name | 技能唯一标识 |
| description | 技能功能描述 |
| input_schema | 输入参数定义 |
| output_schema | 输出结果定义 |
| executor | 执行逻辑 |
| dependencies | 依赖的技能 |

### 7.2 执行流程

```
1. 验证输入参数
2. 检查依赖技能是否满足
3. 执行技能逻辑
4. 验证输出结果
5. 返回执行结果
```

### 7.3 Skill 类型

| 类型 | 说明 | 示例 |
|------|------|------|
| **生成类** | 内容生成 | 文本生成、代码生成 |
| **分析类** | 信息分析 | 数据分析、文本分析 |
| **转换类** | 格式转换 | 翻译、重写 |
| **检索类** | 信息检索 | 搜索、查询 |
| **执行类** | 外部操作 | API调用、命令执行 |

## 八、Tool 详细设计

### 8.1 Tool 接口定义

| 字段 | 类型 | 说明 |
|------|------|------|
| name | str | 工具名称 |
| description | str | 工具功能描述 |
| parameters | JSON Schema | 参数定义 |
| returns | JSON Schema | 返回值定义 |
| timeout | int | 超时时间（秒） |

### 8.2 Tool 类型

| 类型 | 说明 | 示例 |
|------|------|------|
| **Function** | 函数调用 | 计算器、数据处理 |
| **API** | 外部 API | 搜索、天气、数据库 |
| **Database** | 数据库操作 | SQL 查询、向量检索 |
| **File** | 文件操作 | 读取、写入、搜索 |
| **Code** | 代码执行 | Python 脚本执行 |

### 8.3 调用流程

```
1. 参数验证
2. 构建调用请求
3. 执行调用
4. 处理响应
5. 返回结果
```

### 8.4 工具选择策略

| 策略 | 说明 |
|------|------|
| **LRU** | 最近最少使用 |
| **Cost-based** | 按调用成本选择 |
| **Capability-based** | 按能力匹配选择 |
| **Hybrid** | 综合多种策略 |
