# 观察系统 (Observability)

> Agent 的眼睛——感知环境与任务状态，完整记录执行轨迹，采集性能与质量指标。

---

## 一句话定义

**观察系统是 Harness 的日志系统**——负责状态感知、日志记录、指标采集，但不是所有状态都值得看（成本控制）。

---

## 核心概念

| 概念 | 说明 |
|------|------|
| **状态感知** | 感知环境与任务状态 |
| **日志记录** | 完整执行轨迹追踪 |
| **指标采集** | 性能/质量/成本度量 |
| **成本控制** | 不是所有状态都值得看 |
| **观测驱动控制** | 基于观测自动触发控制动作 | L3 核心能力 |

---

## 观察维度

### 1. 环境状态

> 感知代码仓库和 CI/CD 环境的状态

**关键属性**：
- 当前分支
- 已修改文件列表
- 未提交变更
- 测试状态（PASS/FAIL/UNKNOWN）
- CI 状态（RUNNING/SUCCESS/FAILED）

### 2. 任务状态

> 跟踪任务执行进度

**关键属性**：
- 任务 ID
- 当前阶段（PLANNING/EXECUTING/VALIDATING/DONE）
- 进度百分比
- 工具调用列表
- 执行结果列表
- 错误列表

### 3. Agent 状态

> 监控 Agent 运行状态

**关键属性**：
- Agent ID
- 会话 ID
- 当前上下文 Token 数
- 最大上下文 Token 数
- 活跃工具列表
- 最后活动时间

## 日志层级

| 级别 | 内容 | 用途 |
|------|------|------|
| **DEBUG** | 详细执行细节 | 开发调试 |
| **INFO** | 关键事件 | 运行时追踪 |
| **WARNING** | 边缘情况 | 预警 |
| **ERROR** | 失败事件 | 问题排查 |
| **FATAL** | 系统级崩溃 | 立即告警 |

## 指标采集

> **实现说明**：观察系统各模块采用合并实现，代码位于对应的`__init__.py` 文件中：
> - `observability/monitoring/__init__.py` - 监控功能
> - `observability/metrics/__init__.py` - 指标采集
> - `observability/events/__init__.py` - 事件流
> - `observability/alerts/__init__.py` - 告警管理

## 成本控制策略

> 观察系统不是所有状态都值得看，需要成本控制

**控制策略**：
- **优先级过滤**：低于最小优先级的事件不观察
- **采样**：对重复类型事件进行采样
- **聚合**：对相似事件进行聚合处理

### 优先级定义

| 优先级 | 事件类型 | 示例 |
|--------|---------|------|
| **P0 (最高)** | 错误、安全事件 | ERROR, 危险命令拦截 |
| **P1 (高)** | 工具调用、任务完成 | ToolCall, TaskComplete |
| **P2 (中)** | 状态变更 | PhaseChange |
| **P3 (低)** | 详细日志 | DEBUG 级别 |
| **P4 (最低)** | 重复事件 | 相同 tool 连续调用 |

## 观测驱动控制 (Observability-Driven Control)

> L3 核心能力：基于实时观测数据，自动触发控制动作，形成闭环自动化。

### 核心概念

| 概念 | 说明 |
|------|------|
| **观测驱动控制** | 基于实时观测数据，自动触发控制动作 |
| **控制回路** | 观测 → 分析 → 决策 → 动作 → 再观测 |
| **闭环自动化** | 无需人工介入的自动修复与降级 |
| **分级响应** | 根据风险等级采取不同动作 |

### 控制回路架构

```
┌─────────────────────────────────────────────────────────────────┐
│                      观测驱动控制回路                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐    │
│   │  观测   │───▶│  分析   │───▶│  决策   │───▶│  动作   │    │
│   │ 采集    │    │ 模式识别 │    │ 规则引擎 │    │ 执行    │    │
│   └─────────┘    └─────────┘    └─────────┘    └─────────┘    │
│        ▲                                              │         │
│        └──────────────────────────────────────────────┘         │
│                          反馈闭环                               │
└─────────────────────────────────────────────────────────────────┘
```

### 控制规则定义

#### 性能控制规则

| 观测条件 | 触发动作 | 优先级 |
|---------|---------|--------|
| Token 使用 > 80% 预算 | 触发上下文压缩 | P1 |
| Token 使用 > 95% 预算 | 强制截断 + 警告 | P0 |
| 单次响应 > 30s | 降级到低推理模型 | P2 |
| 连续 5 次响应 > 10s | 切换模型路由 | P1 |

> **详细说明**：上下文压缩策略见 [Context 管理](./context.md)

#### 质量控制规则

| 观测条件 | 触发动作 | 优先级 |
|---------|---------|--------|
| 工具调用失败率 > 20% | 暂停 Agent，请求确认 | P0 |
| 连续 3 次相同工具失败 | 标记工具为不可用 | P1 |
| Lint 通过率 < 80% | 触发 Stop Hook 强制修复 | P1 |
| 测试通过率 < 70% | 禁止进入下一阶段 | P0 |

#### 成本控制规则

| 观测条件 | 触发动作 | 优先级 |
|---------|---------|--------|
| 单任务成本 > 预算 80% | 降级到低推理模型 | P1 |
| 单任务成本 > 预算 100% | 暂停任务，请求人类确认 | P0 |
| 高推理模型调用占比 > 30% | 重新规划任务分解 | P2 |

#### 循环控制规则

| 观测条件 | 触发动作 | 优先级 |
|---------|---------|--------|
| Agent 循环 > 10 轮 | 触发 Stop Hook 强制验证 | P1 |
| Agent 循环 > 15 轮 | 中断 + 请求人类介入 | P0 |
| 连续 3 轮无工具调用 | 注入反思 Prompt | P2 |
| 连续 3 轮相同输出 | 强制重置状态 | P1 |

### 上下文焦虑检测

> Anthropic 发现的"上下文焦虑"现象——模型会在自认为接近上下文上限时，开始过早地"收尾"工作

| 观测信号 | 阈值 | 动作 |
|---------|------|------|
| 连续 3 轮输出字符数递减 | 触发 | 注入"还有足够空间"提示 |
| 主动提及"上下文快满了" | 检测到 | 触发上下文压缩，而非重置 |
| 输出中出现"由于上下文限制" | 检测到 | 触发上下文重置 + 生成交接文档 |
| 拒绝执行新任务 | 触发 | 强制上下文重置 + 交接文档 |

## 轨迹反哺机制

> **Harness 不仅要"干活"，还要记录高质量执行轨迹，反哺模型迭代。**
> **竞争优势从模型参数转向执行轨迹数据。**

### 为什么轨迹是竞争优势？

| 维度 | 说明 |
|------|------|
| **第100步后偏差** | Harness 精确检测模型在长任务中的偏差，这些数据反馈到训练 |
| **真实场景数据** | 执行轨迹来自真实用户场景，比静态数据集更有价值 |
| **模型优化闭环** | 轨迹数据 → 训练下一代模型 → 新模型内生化能力 → Harness演进 |

### 轨迹类型与价值

| 轨迹类型 | 数据价值 | 用途 |
|---------|---------|------|
| **成功轨迹** | 高质量示范 | 用于 RLHF 训练 |
| **失败轨迹** | 负样本 | 用于对齐训练 |
| **边界案例** | 长尾场景 | 用于泛化训练 |
| **修复轨迹** | 问题→解决方案 | 用于策略学习 |

### 数据飞轮效应

```
高质量 Harness 设计
    ↓
高质量执行轨迹
    ↓
模型能力提升
    ↓
新能力被内生化
    ↓
 Harness 设计演进
    ↓
（循环）
```
