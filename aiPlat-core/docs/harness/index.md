## 关键设计原理

### 1. 执行循环 - Query → Plan → Act → Validate

**解决的问题**：
- 长任务链中的"静默失败"问题
- 20步任务，单步95%成功率，整体仅36%

**实现（As-Is）**：
- ReAct 执行循环：Reasoning → Acting → Observing（主路径：`execution/loop.py`）
- Plan-Execute：规划 + 顺序执行（主路径：`execution/loop.py`）
- “Ralph Loop/更严格验证模式”若存在，仅作为 To-Be 概念；需以代码实现为准标注状态（避免写成已实现）。

**Hook 拦截点**：

| 阶段 | 拦截点 | 说明 |
|------|--------|------|
| 循环前 | PreLoop | 初始化状态 |
| 思考前 | PreReasoning | 准备思考 |
| 思考后 | PostReasoning | 验证推理 |
| 行动前 | PreAct | 准备工具调用 |
| 行动后 | PostAct | 验证工具结果 |
| 观察前 | PreObserve | 准备观察 |
| 观察后 | PostObserve | 处理观察结果 |
| 循环后 | PostLoop | 清理和保存 |
| 任务完成前 | Stop | 强制验证 |
| 会话开始时 | SessionStart | 状态恢复 |
| 审批前 | PreApprovalCheck | 审批请求前检查 |
| 审批后 | PostApprovalCheck | 审批结果后处理 |
| 合约检查前 | PreContractCheck | 合约范围检查前 |
| 合约检查后 | PostContractCheck | 合约范围检查后 |
| 范围审查 | ScopeReview | 超出范围时的审查 |

### 2. 心跳监控自动注册

**解决的问题**：
- 实时监控 Agent 的运行状态
- 检测卡死、异常等问题
- 及时发现性能下降

**状态判定规则**：

| 状态 | 判定条件 | 说明 |
|------|----------|------|
| 活跃 | 最后活动时间 < 60秒 | 正常运行 |
| 空闲 | 60秒 ≤ 最后活动时间 < 5分钟 | 暂时空闲 |
| 警告 | 5分钟 ≤ 最后活动时间 < 10分钟 | 可能存在问题 |
| 错误 | 10分钟 ≤ 最后活动时间 < 15分钟 | 明显异常 |
| 停滞 | 最后活动时间 ≥ 15分钟 | 可能已卡死 |

**健康分数计算**：
- 根据状态类型确定基础分数（活跃1.0、空闲0.8、警告0.5、错误0.2、停滞0.0）
- 分数范围 0.0 ~ 1.0，低于 0.5 自动告警

### 3. 多 Agent 协调 - 6 种协作模式

| 模式 | 说明 | 适用场景 |
|------|------|----------|
| **Pipeline** | 流水线模式，顺序处理 | 流程化任务 |
| **Fan-out/Fan-in** | 并行处理，结果汇总 | 可分解的并行任务 |
| **Expert Pool** | 动态选择最合适的 Agent | 多种专业能力 |
| **Producer-Reviewer** | 生成+审核，二次验证 | 质量要求高 |
| **Supervisor** | 中心化调度 | 统一调度的复杂任务 |
| **Hierarchical** | 层级任务拆解 | 大型项目分解 |

> **详细说明**：TriAgent 模式（Planner-Generator-Evaluator）的完整说明见 [Evaluator Architecture](#95-evaluator-architecture评估器架构)。

### 4. 观测驱动控制（Observability-Driven Control）

> L3 核心能力：基于实时观测数据，自动触发控制动作，形成闭环自动化。

**控制回路**：
```
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│  观测   │───▶│  分析   │───▶│  决策   │───▶│  动作   │
│ 采集    │    │ 模式识别 │    │ 规则引擎 │    │ 执行    │
└─────────┘    └─────────┘    └─────────┘    └─────────┘
      ▲                                              │
      └──────────────────────────────────────────────┘
                        反馈闭环
```

**控制规则示例（As-Is 最小闭环）**：

| 观测条件 | 触发动作 | 优先级 |
|---------|---------|--------|
| Token 使用 > 80% 预算 | 触发上下文压缩 | P1 |
| 工具调用失败率 > 20% | 暂停 Agent，请求确认 | P0 |
| Agent 循环 > 10 轮 | 触发 Stop Hook 强制验证 | P1 |
| 连续 3 次相同工具失败 | 标记工具为不可用 | P1 |

## 设计补全（Round2）：观测驱动控制（最小闭环规范）

> 目标：把“观测驱动控制”从概念描述落到可实现、可验收、可运维的闭环。

### 1) 最小闭环定义

**采集（Observability）** → **规则评估（Policy）** → **动作执行（Control Action）** → **可观测输出（Events/Logs/Metrics）**

### 2) 指标定义（必须明确来源）

| 指标 | 定义 | 建议采集点 |
|---|---|---|
| tool_error_rate | 最近 N 次工具调用失败率 | ToolResult + HookPhase.POST_TOOL_USE |
| token_budget_used_ratio | 已用 token / 总预算 | LoopState.used_tokens / LoopConfig.max_tokens |
| loop_rounds | 当前 loop 轮数 | LoopState.step_count |

### 3) 规则与动作（最低可用集）

| 条件 | 动作 | 级别 | 备注 |
|---|---|---|---|
| tool_error_rate > 0.2（N>=10） | pause + require_manual | P0 | 输出“为何暂停/需要确认” |
| token_budget_used_ratio > 0.8 | compact_context | P1 | 调用 Memory/Context 压缩策略 |
| loop_rounds > max_steps | stop | P1 | 触发 STOP Hook 并写审计 |

### 4) 接线位置（单一真相）

建议接线顺序：
1. Loop/Graph 产出观测事件（POST_* hooks）
2. PolicyEngine 评估规则（可配置）
3. BaseLoop/Executor 执行动作（pause/compact/stop/deny_tool）

### 5) 验收标准（必须可复现）

1. 构造工具连续失败：达到阈值后，loop 必须暂停并返回 require_manual 语义
2. 构造 token 超预算：必须触发 compact_context，并在事件中记录压缩前后长度变化
3. 构造循环超限：必须触发 stop，并输出 stop reason

### 5. 自我进化机制

**解决的问题**：
- Agent 需要根据执行历史自动优化策略
- 静态配置无法适应动态环境

**触发条件**：

| 条件 | 说明 | 默认值 |
|------|------|--------|
| 最小样本 | 需要积累足够执行数据 | 10 次 |
| 性能阈值 | 成功率低于此值触发优化 | 70% |
| 进化间隔 | 避免频繁进化 | 1 小时 |

**进化内容**：
- 参数调整：温度、top_p、max_tokens 等 LLM 参数
- 策略切换：调整工具选择、记忆检索策略
- 能力扩展：注册新技能、启用新工具

### 6. 三层反馈循环系统

**解决的问题**：
- 优化策略的安全性问题
- 如何在不影响生产的前提下验证优化

**层级说明**：

| 层级 | 说明 | 触发条件 |
|------|------|----------|
| **LOCAL** | 本地反馈，快速实验 | 开发/测试环境 |
| **PUSH** | 配置中心，版本管理 | 性能提升 > 10% |
| **PROD** | 生产环境，全局生效 | 通过灰度验证 |

**上升机制**：
- LOCAL 层验证改进后，推送到 PUSH 层
- PUSH 层验证性能达标后，推送到 PROD 层

**下降机制**：
- PROD 层检测异常，自动回滚到上一版本
- PUSH 层可手动回滚到任意历史版本
- LOCAL 层自动回滚，无需人工介入

### 7. 轨迹反哺机制

> Harness 不仅要"干活"，还要记录高质量执行轨迹，反哺模型迭代。

**轨迹类型与价值**：

| 轨迹类型 | 数据价值 | 用途 |
|---------|---------|------|
| **成功轨迹** | 高质量示范 | 用于 RLHF 训练 |
| **失败轨迹** | 负样本 | 用于对齐训练 |
| **边界案例** | 长尾场景 | 用于泛化训练 |

**数据飞轮效应**：
```
高质量 Harness 设计
    ↓
高质量执行轨迹
    ↓
模型能力提升
    ↓
新能力被内生化
    ↓
Harness 设计演进
    ↓
（循环）
```

### 8. 三层管理器分离

| 管理器 | 职责 | 数据生命周期 |
|--------|------|--------------|
| **配置管理器** | 管理 Agent 运行参数 | 持续有效 |
| **性能追踪器** | 记录任务执行指标 | 最近1000条 |
| **历史管理器** | 维护执行轨迹 | 最近100条 |

## 九、Harness Engineering 核心原则

### 9.1 Sprint Contract（冲刺合约）

> **核心理念**：在开发前达成明确的功能范围和验收标准协议，确保需求理解一致，减少返工。

**解决的问题**：
- 需求不清晰导致的无限循环
- 验收标准不明确导致的交付争议
- 开发过程中范围蔓延（Scope Creep）

**合约结构**：

| 字段 | 说明 | 示例 |
|------|------|------|
| **功能范围** | 明确要实现的功能 | "实现用户登录功能，包括用户名密码登录和验证码登录" |
| **验收标准** | 可量化的交付条件 | "登录成功率 ≥ 99%，响应时间 < 200ms" |
| **时间约束** | 明确的迭代周期 | "1个Sprint（2周）" |
| **依赖声明** | 外部依赖和前置条件 | "依赖用户中心提供用户数据API" |
| **风险预案** | 已知风险和应对策略 | "若第三方API不可用，则使用本地缓存" |

**合约工作流**：

```
需求输入 → 范围澄清 → 验收标准定义 → 风险评估 → 合约签署 → 开发执行 → 验收交付
                                                                 │
                                                        若验收不通过 → 修复或重新评估范围
```

**在 Harness 中的实现**：
- Agent 启动时创建 ContractContext
- 每个执行步骤检查是否在合约范围内
- 验收标准作为 Stop Hook 的验证条件
- 超出范围时触发 ScopeReview 流程

### 9.2 File-based Communication（文件化通信）

> **核心理念**：通过结构化文件进行 Agent 间通信，确保信息传递的准确性和可追溯性。

**解决的问题**：
- 多 Agent 协作时的信息丢失
- 任务进度和状态不透明
- 交接过程的信息失真

**核心文件类型**：

| 文件类型 | 用途 | 内容 |
|---------|------|------|
| **spec.md** | 需求规格说明 | 功能需求、性能指标、边界条件 |
| **sprint-report.md** | 冲刺报告 | 完成情况、问题、待办 |
| **feedback.md** | 反馈记录 | 问题描述、影响范围、修复建议 |
| **handoff.md** | 交接文档 | 当前状态、后续任务、注意事项 |
| **review.md** | 评审记录 | 评审意见、修改要求、通过状态 |

**文件流转示例**：

```
┌─────────────┐    spec.md    ┌─────────────┐
│  Planner    │ ──────────────▶│ Generator  │
└─────────────┘                └──────┬──────┘
                                       │ code + review.md
                                       ▼
                              ┌─────────────┐
                              │  Evaluator  │
                              └──────┬──────┘
                                     │ feedback.md
                                     ▼
                              ┌─────────────┐
                              │    Loop     │ (根据反馈调整)
                              └─────────────┘
```

**在 Harness 中的实现**：
- ContextService 管理文件生命周期
- Agent 协作时自动生成/读取相关文件
- 文件版本控制和差异对比
- 支持文件模板和自动填充

### 9.3 Value Decay Curve（价值衰减曲线）

> **核心理念**：不同类型的反馈随着时间推移价值衰减速度不同，需要区别对待。

**价值衰减速率**：

```
价值
│
│  Format Affinity (格式亲和性)
│  ╲
│   ╲________ 衰减最快
│    ╲
│     ╲  Capability Complement (能力互补)
│      ╲________ 衰减中等
│       ╲
│        ╲   Feedback Quality (反馈质量)
│         ╲________ 衰减最慢
│          ╲
│           ╲___________________________ 时间
```

| 反馈类型 | 衰减速率 | 原因 | 应用策略 |
|---------|---------|------|----------|
| **Format Affinity** | 最快 | 模型对格式敏感，格式变化导致参考价值下降 | 及时转换为通用格式 |
| **Capability Complement** | 中等 | 能力边界随模型升级变化 | 定期更新能力映射 |
| **Feedback Quality** | 最慢 | 核心反馈原则长期有效 | 长期积累，形成知识库 |

**在 Harness 中的实现**：
- 轨迹数据标注价值衰减类型
- 自动触发格式转换（将 Format Affinity 转为通用格式）
- 定期评估能力映射的有效性
- 反馈质量数据进入长期知识库

### 9.4 Format Affinity（格式亲和性）

> **核心理念**：模型对特定输出格式有偏好，高格式亲和性意味着更好的执行效果。

**格式亲和性维度**：

| 维度 | 说明 | 影响 |
|------|------|------|
| **结构亲和性** | 对特定输出结构的偏好 | JSON > Markdown > 纯文本 |
| **风格亲和性** | 对语言风格的偏好 | 简洁 > 冗长、正式 > 口语 |
| **长度亲和性** | 对输出长度的偏好 | 适中对齐模型上下文窗口 |
| **示例亲和性** | 对 Few-shot 示例的响应 | 有示例 > 无示例 |

**格式优化策略**：

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| **格式锁定** | 固定输出格式，减少变化 | 稳定性要求高的任务 |
| **格式渐进** | 从简单到复杂递进 | 复杂任务，分步处理 |
| **格式反馈** | 根据执行结果调整格式 | 需要优化的任务 |
| **格式模板** | 提供标准化模板 | 批量任务，保持一致 |

**在 Harness 中的实现**：
- ModelService 分析模型的格式偏好
- PromptService 自动适配最佳格式
- 格式亲和性纳入轨迹评估指标
- 支持格式配置的动态调整

### 9.5 Evaluator Architecture（评估器架构）

> **核心理念**：Planner-Generator-Evaluator 三智能体架构，通过评估器实现动态验证和迭代优化。

**TriAgent 架构详解**：

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│   Planner   │───▶│  Generator  │───▶│  Evaluator  │
│   需求扩展   │    │   增量实现   │    │   动态测试   │
└─────────────┘    └─────────────┘    └─────────────┘
      │                  │                  │
      ▼                  ▼                  ▼
  ┌─────────┐       ┌─────────┐       ┌─────────┐
  │需求分析  │       │代码生成  │       │结果验证  │
  │任务拆解  │       │测试编写  │       │质量评估  │
  │计划制定  │       │文档更新  │       │反馈生成  │
  └─────────┘       └─────────┘       └─────────┘
```

| Agent | 职责 | 输入 | 输出 |
|-------|------|------|------|
| **Planner** | 需求理解和任务规划 | 用户需求、spec.md | 任务列表、执行计划 |
| **Generator** | 增量实现和自测 | 任务列表、plan | 代码、单元测试 |
| **Evaluator** | 结果验证和质量评估 | 代码、测试用例 | 验证报告、反馈 |

**Evaluator 评估维度**：

| 维度 | 指标 | 阈值 |
|------|------|------|
| **正确性** | 测试通过率 | ≥ 95% |
| **性能** | 响应时间、吞吐量 | 符合 spec 要求 |
| **可维护性** | 代码复杂度、重复度 | 符合团队标准 |
| **安全性** | 漏洞扫描、权限检查 | 无高危漏洞 |

**循环机制**：
- Generator 输出 → Evaluator 评估 → 通过? → 结束
- 不通过 → 反馈给 Generator → 重新实现 → 再次评估
- 最大循环次数：3次，超过则升级处理

### 9.6 Build to Delete（设计目标）

> **核心理念**：随着模型能力提升，Harness 应逐步"瘦身"，删除不必要的约束。

**核心观点**：
- **模型决定 Agent 的上限，Harness 决定 Agent 的下限**
- 好的 Harness 设计应该能随着模型进化而简化
- 为 Opus 4.5 设计的厚重 Harness，在 Opus 4.6 上可能被大幅简化

**实践原则**：
- 定期审视当前 Heavy Harness 在新模型上是否过于复杂
- 优先使用"软约束"（Hook），而非"硬编码"（强制规则）
- 保留可删除的"可选层"，而非不可逆的"必选层"
