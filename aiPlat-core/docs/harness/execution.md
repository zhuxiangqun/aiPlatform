## 一句话定义

**执行系统是 Harness 的进程管理器**——负责 Agent 的循环执行、流式响应、重试逻辑和 Token 预算管理。

---

## 核心机制

### 1. 循环控制

> **执行循环接口**
> 
> 定义 Agent 执行循环的标准接口，包含以下核心方法：
> - **运行循环**：启动 Agent 的执行循环，管理整个执行过程
> - **继续判断**：根据当前状态判断是否继续循环执行

### 2. 执行阶段

```
┌─────────────────────────────────────────────────────────────┐
│                     Agent Loop                               │
├─────────────────────────────────────────────────────────────┤
│  思考 (Reasoning)  →  行动 (Acting)  →  观察 (Observing)  │
│         ↑                     ↑                     ↓         │
│         └─────────────────────┴─────────────────────┘       │
│                    循环直到完成或停止                        │
└─────────────────────────────────────────────────────────────┘
```

### 3. 执行器类型

| 执行器 | 说明 | 适用场景 |
|--------|------|----------|
| **UnifiedExecutor** | 统一执行器，处理大多数任务 | 常规任务 |
| **SandboxExecutor** | 沙箱执行器，隔离危险操作 | 代码执行、文件操作（To-Be） |
| **BackgroundTasks** | 后台任务处理 | 耗时任务（To-Be） |

> **实现状态补充**：
> - `UnifiedExecutor`：代码存在（`core/harness/execution/executor/unified.py`），但默认 executors 注册逻辑仍需补齐
> - `SandboxExecutor/BackgroundTasks`：当前为设计占位（To-Be），需明确接线方式与验收

## 设计补全（Round2）：执行器与 Hook 的可验收契约

### 1) 执行器注册闭环

必须定义：
- 默认注册哪些 executor（unified、langgraph、sandbox 等）
- 选择策略（按 task/tool 风险、按配置、按策略引擎）
- fallback 条件（langgraph 不可用时降级）

### 2) 必备验收用例

1. 启动后，能通过接口/日志确认默认 executor 已注册（而不是空 pass）
2. 配置切换 executor 后，实际执行路径发生变化（可被测试验证）


### 4. 重试机制

> **统一重试管理器**
> 
> 提供可配置的重试策略：
> - **最大重试次数**：默认3次
> - **退避因子**：指数退避，默认2.0
> - **重试条件**：可配置哪些异常需要重试

### 5. Token 预算管理

> **执行策略 - Token 预算控制**
> 
> 控制执行过程中的 Token 消耗：
> - **总预算上限**：默认 100,000 tokens
> - **推理预算上限**：默认 60,000 tokens
> - **预算检查**：每次执行前检查剩余预算

## Hook 拦截点

| 阶段 | 拦截点 | 说明 |
|------|--------|------|
| 循环前 | PreLoop | 初始化状态 |
| 思考前 | PreReasoning | 准备思考 |
| 思考后 | PostReasoning | 验证推理 |
| 行动前 | PreAct | 准备工具调用 |
| 行动后 | PostAct | 验证工具结果 |
| 观察前 | PreObserve | 准备观察 |
| 观察后 | PostObserve | 处理观察结果 |
| 循环后 | PostLoop | 清理和保存 |
| 任务完成前 | Stop | 强制验证 |
| 会话开始时 | SessionStart | 状态恢复 |

## 耐久性问题 (Durability Problem)

> 静态榜单无法捕捉长任务链中的问题——这是 Harness 存在的核心原因之一。

### 问题描述

| 维度 | 静态榜单 | 长任务链 |
|------|---------|---------|
| 模型表现 | 1-2次尝试解决难题 | 需要执行数百次工具调用 |
| 检测能力 | 容易衡量 | 榜单上1%分数差异无法捕捉 |

## 证据索引（Evidence Index｜抽样）

- Loop-first 执行主路径（As-Is）：`core/harness/execution/loop.py`
- Agent 委托 Loop 执行：`core/apps/agents/base.py: BaseAgent.execute()`
- LangGraph 子系统（To-Be 主编排方向/部分现用）：`core/harness/execution/langgraph/*`
| 失败模式 | 单次推理失败 | 第50步偏离指令、第100次工具调用后失去推理连贯性 |

### 数学说明

```
20步任务，单步成功率95%
无 Harness 时整体成功率 = 0.95^20 ≈ 36%

这就是"静默失败"的致命性。
```

### Harness 如何解决

| 问题 | Harness 解决方案 |
|------|----------------|
| 状态丢失 | 记忆系统持久化状态 |
| 工具失败 | 重试逻辑 + 两振出局 |
| 偏离指令 | Stop Hook 强制验证 |
| 无限循环 | 循环检测 + 人工介入 |
| 推理断裂 | 观测驱动控制 + 预算管理 |

### 关键指标

| 指标 | 说明 | 阈值 |
|------|------|------|
| **长任务成功率** | 50步以上任务的成功率 | >80% |
| **工具调用连续性** | 第100次工具调用的成功率 | >90% |
| **指令跟踪率** | 任务末期仍遵循初始指令 | >95% |

## ReAct 执行循环

> **ReAct 执行流程**
> 
> ReAct（Reasoning + Acting）是最常用的执行模式：
> 
> 1. **推理阶段**：LLM 推理决定下一步行动
> 2. **行动阶段**：如需工具调用，执行工具
> 3. **观察阶段**：处理工具执行结果
> 4. **循环判断**：判断是否继续或结束

### 状态管理

**状态结构设计**：

| 字段 | 类型 | 说明 |
|------|------|------|
| messages | List[Message] | 对话历史 |
| context | Dict | 任务上下文 |
| tools_calls | List[ToolCall] | 待执行的工具调用 |
| results | Dict | 工具执行结果 |
| step_count | int | 已执行步骤数 |
| metadata | Dict | 元信息 |

**状态更新规则**：

- 每个 Node 可以读取和写入 State
- State 更新是原子性的，保证一致性
- 支持中间状态检查点，用于恢复

### LangGraph 与 Harness 的关系

| LangGraph 概念 | Harness 实现 | 说明 |
|----------------|-------------|------|
| **Node** | Agent / Skill / Tool | 执行步骤的原子单元 |
| **Edge** | Workflow / Condition | 节点间的执行顺序和跳转 |
| **State** | AgentContext / WorkflowContext | 节点间传递的上下文 |
| **Cycle** | Loop / Multi-turn | 迭代执行和多轮对话 |
