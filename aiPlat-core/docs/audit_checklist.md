# 设计文档一致性审计 Checklist（L1/L2 双层｜设计真值治理工具）

> **设计真值口径**：本 checklist 用于持续约束“文档宣称”与“代码事实”的一致性；关键断言需可追溯（代码入口/测试/命令）。  
> 适用范围：`aiPlat-core/docs/**`、`aiPlat-infra/docs/**`、以及仓库根目录 `reports/**`（审计产物与历史快照）。

> 目的：避免“文档宣称 ≠ 代码事实”“实现可运行但语义不完整/不可运维”等问题在后续迭代反复出现。  
> 使用方式：每个迭代（或每次大改）至少跑一次 **L1 一致性审计**；发布前/架构升级时跑 **L2 语义与可用性审计**。

---

## 0. 统一原则（本 checklist 的硬规则）

1) **每条设计声明必须可追溯**（二选一或多选）：
   - 代码入口：文件路径 + 关键类/函数名
   - 验证方式：单测/集成测试/命令/可观测指标
2) **默认行为必须可用**：若设计引入权限/安全/回滚/治理等控制，必须说明并验证默认策略，不允许“默认全拒绝/默认无效”导致开箱即不可用。
3) **文档与代码的“存在性表述”必须准确**：
   - 代码不存在 → ❌
   - 代码存在但未接通 → 🔧
   - 代码存在但语义缺失/误导 → 🚨
4) **跨层边界必须唯一**：同一种能力（例如 MCP、权限、状态机）不允许出现两套“看似都在用”的实现并存，除非明确写出主/备、迁移计划与禁用路径。

---

## 1. L1：一致性审计（Doc ↔ Code 是否对齐）

### 1.1 文档清单与主入口
- [ ] 本次审计覆盖哪些 docs（core/infra/management/根架构）？记录清单与版本/日期。
- [ ] 每个子系统文档（Harness/Agent/Skill/Tool/Infra）是否都包含：
  - [ ] 模块定位
  - [ ] 模块边界/依赖方向
  - [ ] 关键接口与数据模型
  - [ ] 关键流程（至少一张流程图或伪代码）

### 1.2 “设计声明 → 代码入口”映射
对每个模块至少抽查 10 条关键声明：
- [ ] 声明是否能定位到具体文件/函数？
- [ ] 入口是否在实际执行路径可达（不是“存在但从未被调用”）？
- [ ] 若声明涉及工厂映射/注册表（create_* / register_*），是否与文档清单一致？

### 1.3 文档结构/路径正确性
- [ ] 文档中的目录树/文件名是否真实存在？
- [ ] 文档示例 import 路径是否能在代码中找到对应模块？
- [ ] 文档中的“存在/不存在”结论是否与仓库一致（禁止凭印象）？

### 1.4 跨项目边界一致性（core ↔ infra ↔ management）
重点检查：
- [ ] 同一能力在不同 repo 的归属是否一致（例如 MCP、Observability、DI、权限）？
- [ ] 是否出现 “infra 文档写 core 调用，但 core 实际自实现” 的偏差？
- [ ] 关键 API（management→core）字段/语义是否一致（至少抽查 3 条 endpoint）？

---

## 2. L2：语义与可用性审计（实现是否合理/完整/可运维）

### 2.1 权限/安全/治理的“默认闭环”
如果系统存在 PermissionManager / PolicyEngine / 审批 / 合约检查：
- [ ] 默认账号（system/admin）是否具备最小可用权限？（开箱不 403）
- [ ] 是否存在授权闭环：grant/revoke/list 的 API 或可配置策略？
- [ ] 权限检查是否在正确层级执行（入口、Tool/Skill/Agent、或统一 wrapper）？是否重复/遗漏？
- [ ] 若默认拒绝，文档必须明确“如何启用/如何授予”，并有验收用例。

### 2.2 版本管理/回滚语义
如果宣称版本/回滚：
- [ ] 回滚是否影响后续行为（不仅仅是 is_active 标记）？
- [ ] 回滚接口是否校验成功并返回当前 active_version？
- [ ] 是否有“版本创建→查询→回滚→再执行”的集成测试？

### 2.3 Harness 执行链路闭环（Loop/Graph/Executor）
- [ ] 执行主路径是否唯一清晰（Loop vs LangGraph 的主/备关系写明）？
- [ ] fallback 路径是否可用、触发条件是否明确？
- [ ] 若文档宣称 Executor/Sandbox/BackgroundTasks，是否有接线与验收？若未完成需标注为 🔧。

### 2.4 Hook 系统可达性（安全/审批/合约/会话）
如果 HookPhase 定义了治理阶段：
- [ ] 执行主路径是否真的触发这些 HookPhase（SESSION_START/END、APPROVAL、CONTRACT、SCOPE_REVIEW、STOP）？
- [ ] 默认 hook 集合是否包含最低安全钩子（例如 TokenLimit/SecurityScan）？
- [ ] 至少有 2-3 个单测验证 “触发 → hook 执行 → 产生可观测结果/阻断”。

### 2.5 状态模型统一（Agent/Task/Lifecycle）
- [ ] 是否存在多套状态枚举并存？（interfaces/harness/management/registry）
- [ ] 是否有“唯一真相”状态枚举？其他层是否提供映射且有校验？
- [ ] API 返回的 status 字段是否稳定、可被前端/运维依赖？

### 2.6 可观测性与“观测驱动控制”
如果文档宣称 observability-driven control：
- [ ] 指标采集在哪里？（metrics/logs/events）
- [ ] 规则评估在哪里？（policy/hook/controller）
- [ ] 动作执行在哪里？（pause/compact/deny_tool/降级）
- [ ] 是否有稳定验收（触发阈值 → 动作发生 → 可观测）？

## 3. Round2 open 项如何被本 checklist 捕获（复盘映射）

> 这部分用于解释“为何第一轮易漏、第二轮能发现”，并指向 checklist 的具体条目。

- **R2-AUTH-001（权限默认全拒绝）**  
  - 第一轮易漏：看到有 `check_permission` 与 PermissionManager 就认为“能力已落地”，未审默认策略与授权闭环。  
  - checklist 捕获点：L2-2.1（默认闭环/开箱可用/授权 API）。

- **R2-SKILL-VERSION-SEM-001（回滚语义缺失）**  
  - 第一轮易漏：聚焦“返回结构/API 一致性”，未验证“回滚是否改变行为”。  
  - checklist 捕获点：L2-2.2（回滚必须影响行为 + 集成测试）。

- **R2-ARCH-BOUNDARY-001（MCP 边界/依赖方向偏差）**  
  - 第一轮易漏：以 core 为中心审计，未对 infra 文档的“依赖方向声明”做 cross-repo 对照。  
  - checklist 捕获点：L1-1.4（跨 repo 边界一致性）。

- **R2-HOOK-SEC-001（治理 Hook 未接通）**  
  - 第一轮易漏：看到 HookPhase 定义齐全/Hook 类存在，未验证“执行路径可达”。  
  - checklist 捕获点：L2-2.4（HookPhase 触发链路 + 单测）。

- **R2-AGENT-STATE-001（状态模型割裂）**  
  - 第一轮易漏：优先修“功能可跑”问题，状态统一属于治理/运维层的语义问题。  
  - checklist 捕获点：L2-2.5（唯一枚举与映射校验）。

- **R2-TOOL-THREADSAFE-001（线程安全/追踪承诺不足）**  
  - 第一轮易漏：实现已从“无”到“有 wrapper”，但未进一步核对文档承诺的锁/trace 是否真实使用。  
  - checklist 捕获点：L2-2.1/L2-2.6 与 L1-1.2（承诺→实现细节验证）。

- **R2-DOC-PATH-001 / R2-HARN-EXEC-001 / R2-INFRA-MCP-IMPL-001（文档结构/存在性错误）**  
  - 第一轮易漏：未做系统性“路径存在性/文件清单”校验。  
  - checklist 捕获点：L1-1.3（文档路径/存在性校验）。

- **R2-OBS-CONTROL-001（观测驱动控制缺闭环）**  
  - 第一轮易漏：属于“设计完整性/验收标准”问题，不会在代码断裂修复中自然暴露。  
  - checklist 捕获点：L2-2.6（采集→规则→动作闭环 + 验收）。
