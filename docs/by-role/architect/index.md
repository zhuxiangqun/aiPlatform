# 👷 架构师指南

> aiPlatform - 架构设计与依赖管理

---

## 🎯 架构师关注点

作为架构师，您需要了解：
- **依赖方向**：如何确保各层之间的单向依赖
- **接口设计**：如何设计清晰的接口
- **扩展性**：如何支持多种实现
- **性能**：如何优化性能和资源使用
- **可追溯性**：如何记录架构决策

---

## 🏗️ 四层架构设计

### 架构概览

aiPlatform 采用四层架构设计：

**Layer 0 - 基础设施层**：
- 提供最底层的基础设施能力
- 完全独立，不依赖任何内部模块
- 提供数据库、LLM、向量存储、配置等基础服务

**Layer 1 - AI中台核心层**：
- 封装 AI 核心能力和业务逻辑
- 提供 Harness 架构、编排引擎、Agent、Skill 等核心能力
- 依赖基础设施层

**Layer 2 - 平台服务层**：
- 对外暴露 API，提供平台级服务
- 提供认证、多租户、限流、计费等平台能力
- 依赖核心层

**Layer 3 - 应用层**：
- 面向用户的应用
- 提供 Gateway、CLI、Web UI 等用户界面
- 依赖平台层

---

## 🔗 依赖管理

### 单向依赖原则

**严格规则**：
- 上层可以依赖下层，下层不能依赖上层
- 只能通过公开接口访问，不能跨层调用

**正确的依赖**：
```
app → platform → core → infra
```

**禁止的依赖**：
- app 直接访问 core（应该通过 platform）
- app 直接访问 infra（应该通过 platform）
- platform 直接访问 infra（应该通过 core）
- 任何下层访问上层

---

### 依赖检查

#### CI 自动化检查

以下检查在 PR 时自动执行：

| 检查项 | 工具 | 违规时 |
|--------|------|--------|
| 层间依赖方向 | import-linter | 阻止合并 |
| 无循环依赖 | dependency-cruiser | 阻止合并 |
| 接口稳定性 | API diff | 需架构师审批 |

#### 检查配置文件

配置文件位于项目根目录：
- `.import-linter.yml`：定义各层允许的依赖
- `.dependency-cruiser.js`：定义模块边界

#### 模块边界

各层对外暴露的唯一入口：

| 层 | 对外入口 | 其他模块不可直接导入 |
|----|----------|---------------------|
| infra | `aiPlat_infra.factories` | 具体实现类 |
| core | `aiPlat_core.facades.CoreFacade` | Agent、Skill 具体类 |
| platform | REST API、GraphQL API | 内部服务类 |
| app | 无（不是 API 层） | 无 |

### ADR-006: 双网关架构分流职责

**状态**：已采纳

**背景**：系统需要同时支持外部消息渠道（Telegram、Slack、WebChat）和 API 调用，两个入口职责不同。

**决策**：采用双网关架构，消息网关（Layer 3）负责渠道适配，API 网关（Layer 2）负责安全治理。

**理由**：
1. 职责分离，消息网关专注渠道适配，API 网关专注安全治理
2. 安全统一，所有请求在 API 网关统一认证和限流
3. 灵活扩展，新渠道只需在消息网关添加适配器
4. 易于测试，两个网关可以独立测试

**架构设计**：
```
外部渠道 → 消息网关(Layer 3) → API 网关(Layer 2) → 核心服务
         │                        │
         │ 渠道适配               │ 认证、限流
         │ 消息格式转换           │ 路由、熔断
         │ 协议转换               │ 负载均衡
```

**替代方案**：
- 单一网关处理所有：职责不清，代码耦合
- 多层网关：增加不必要的复杂度

**后果**：
- 正面：职责清晰，安全策略统一，易于扩展新渠道
- 负面：多一层转发，增加约 1-2ms 延迟

- 决策的**背景**和**后果**
- 决策的**状态**和**历史**

### ADR-001: 采用四层架构而非微服务

**状态**：已采纳

**背景**：系统需要清晰的模块边界，支持 AI 能力的组合和编排，便于测试和部署。

**决策**：采用四层架构设计（infra、core、platform、app），单向依赖。

**理由**：
1. 依赖清晰，单向依赖，无循环依赖
2. 边界明确，各层职责清晰，便于团队分工
3. 易于测试，可以逐层进行单元测试和集成测试
4. 部署灵活，可以整体部署或分层部署

**替代方案**：
- 微服务架构：复杂度高，需要服务发现、熔断等基础设施
- 单体架构：模块边界不清晰，扩展性差

**后果**：
- 正面：模块边界清晰，便于团队分工
- 负面：需要更严格的依赖管理

---

### ADR-002: 使用 LangGraph 作为编排引擎

**状态**：已采纳

**背景**：核心层需要提供 AI Agent 的编排能力，支持复杂的工作流和状态管理。

**决策**：使用 LangGraph 作为核心层的编排引擎。

**理由**：
1. 原生支持状态机，天然支持复杂工作流
2. 与 LangChain 生态无缝集成
3. 支持持久化和检查点
4. 活跃的社区支持

**替代方案**：
- 自研编排引擎：开发成本高，需要长期维护
- Temporal：过于复杂，需要额外部署服务

**后果**：
- 正面：快速实现复杂工作流，支持状态持久化
- 负面：依赖第三方库，需要团队学习新概念

---

### ADR-003: 门面模式隔离各层依赖

**状态**：已采纳

**背景**：各层之间需要交互，但需要避免直接依赖具体实现，保持层间边界清晰。

**决策**：各层通过 Facade 类暴露统一的能力入口。

**理由**：
1. 简化接口，Facade 提供简化的统一接口
2. 隔离变化，内部实现变化不影响调用方
3. 便于测试，可以轻松 Mock Facade 进行测试
4. 清晰边界，明确各层对外的暴露点

**层间 Facade 设计**：
- **app层**：通过 REST API 访问 platform 层
- **platform 层**：通过 API Controller 暴露服务
- **core 层**：通过 CoreFacade 暴露核心能力
- **infra 层**：通过 factories 暴露基础设施实例

**后果**：
- 正面：层间边界清晰，调用方式统一
- 负面：增加了一层抽象，需要维护 Facade 接口

---

### ADR-004: 配置驱动切换 LLM 提供商

**状态**：已采纳

**背景**：系统需要支持多种 LLM 提供商，并能够在不修改代码的情况下切换。

**决策**：采用配置驱动的方式切换 LLM 提供商，通过工厂模式创建实例。

**理由**：
1. 零代码修改，通过配置文件切换提供商
2. 环境隔离，不同环境可以使用不同提供商
3. 便于测试，开发环境可以使用 Mock 提供商
4. 灵活切换，支持运行时动态切换

**支持的提供商**：
- OpenAI（GPT-4、GPT-3.5）
- Anthropic（Claude）
- DeepSeek
- 本地模型（Ollama、vLLM）
- Mock（用于测试）

**后果**：
- 正面：客户可以在不同提供商之间切换，便于 A/B 测试
- 负面：需要维护多个提供商的适配器

---

### ADR-005: 禁止平台层直接访问基础设施层

**状态**：已采纳

**背景**：需要明确各层之间的依赖关系，防止依赖混乱。

**决策**：禁止平台层直接访问基础设施层，所有基础设施能力必须通过核心层间接访问。

**理由**：
1. 保持依赖清晰，单向依赖，便于理解和维护
2. 便于测试，可以在核心层 Mock 基础设施
3. 职责明确，核心层封装基础设施能力，平台层封装业务能力
4. 避免循环依赖

**正确访问路径**：
```
platform → core.CoreFacade → infra.factories
```

**错误访问路径（禁止）**：
```
platform → infra.factories ✗
```

**后果**：
- 正面：依赖关系清晰，职责划分明确，安全策略在单点统一实施
- 负面：平台层需要通过核心层间接访问基础设施，增加约 1-2ms 调用延迟

## 🔌 接口设计原则

### 接口与实现分离

每一层都通过接口抽象，不依赖具体实现：
- 基础设施层定义统一的服务接口
- 核心层定义统一的 Agent、Skill、编排接口
- 平台层定义统一的 API、认证、多租户接口
- 应用层通过 REST API 访问平台能力

---

### 接口隔离原则

每个接口只暴露必要的方法，遵循接口隔离原则：
- 数据库接口只定义数据访问方法
- LLM 接口只定义对话和嵌入方法
- Agent 接口只定义执行方法

---

## 🔧 扩展性设计

### 支持多种实现

通过接口与实现分离，支持多种实现：
- 数据库支持 PostgreSQL、MySQL、MongoDB 等
- LLM 支持 OpenAI、Anthropic、DeepSeek 等
- 向量存储支持 FAISS、Milvus、Pinecone 等

### 配置驱动切换

通过配置文件切换实现，无需修改代码：
- 修改配置文件中的类型字段
- 工厂函数根据类型返回对应实现

## ⚡ 性能优化

### 连接池设计

| 连接类型 | 最小值 | 最大值 | 说明 |
|----------|--------|--------|------|
| 数据库连接池 | 5 | 20 | 根据负载动态调整 |
| Redis 连接池 | 10 | 50 | 支持高并发 |
| HTTP 连接池 | 10 | 100 | 支持 keep-alive |
| 向量数据库连接池 | 5 | 30 | 根据查询频率调整 |

### 缓存策略

| 缓存类型 | 缓存对象 | TTL | 失效策略 |
|----------|----------|-----|----------|
| LLM 响应缓存 | 相同 prompt 的响应 | 1 小时 | 精确匹配 |
| 向量查询缓存 | 相似向量的搜索结果 | 5 分钟 | 写入失效 |
| 配置缓存 | 运行时配置 | 动态 | 配置变更时刷新 |
| 会话缓存 | 用户会话信息 | 30 分钟 | 活动续期 |

### 缓存命中率监控

- LLM 响应缓存命中率：目标 > 60%
- 向量查询缓存命中率：目标 > 80%
- 配置缓存命中率：目标 > 95%

### LLM 调用优化

LLM 调用与普通数据库调用不同，需要特殊优化：

| 优化策略 | 说明 | 配置建议 |
|----------|------|----------|
| 请求超时 | 避免无限等待 | 30-60秒 |
| 重试机制 | 临时故障自动重试 | 最多3次，指数退避 |
| 降级策略 | 主 LLM 不可用时切换备用 | 支持多提供商配置 |
| 流式响应 | 减少首字延迟 | 启用 SSE/WebSocket |
| Token 限制 | 控制成本和响应时间 | max_tokens=4096 |
| 并发控制 | 避免 LLM 提供商限流 | 连接池最大 10 |

**LLM 调用监控指标**：
- 平均响应时间：目标 < 10秒
- Token 使用量：按小时统计
- 成功率：目标 > 99%
- 降级触发次数：目标 < 1次/天

## 📊 架构健康度检查

### 定期检查项

| 检查项 | 频率 | 目标 | 不达标处理 |
|--------|------|------|------------|
| 循环依赖检查 | 每次 PR | 0 个循环 | 阻止合并 |
| 跨层违规导入 | 每次 PR | 0 个违规 | 阻止合并 |
| 接口变更审批 | 每次变更 | 100% 审批 | 记录并审批 |
| ADR 覆盖度 | 季度 | 重要决策有 ADR | 补充 ADR |
| 技术栈版本 | 月度 | 无 EOL 版本 | 升级计划 |
| 测试覆盖率 | 每次 PR | > 80% | 强制要求 |

### 架构评审

**需要架构评审的场景**：
- 引入新的外部依赖
- 修改层间契约（API、Facade 接口）
- 新增重大功能模块
- 性能优化方案影响架构
- 安全策略变更

**评审流程**：
1. 提出架构变更提案（RFC）
2. 架构评审会议讨论
3. 记录决策到 ADR
4. 实施变更

---

### 层间契约

| 依赖关系 | 契约形式 | 变更影响 |
|----------|----------|----------|
| app → platform | OpenAPI 规范 | 接口变更需版本升级 |
| platform → core | CoreFacade 接口 | 接口变更需版本升级 |
| core → infra | 工厂接口 | 新增实现不影响上层 |

### 版本策略

- **主版本号（MAJOR）**：不兼容的接口变更
- **次版本号（MINOR）**：新增功能，向下兼容
- **补丁版本（PATCH）**：bug修复，完全兼容
