# aiPlat 核心能力层全量分析（历史分析快照｜As-Is/To-Be 对照）

> **历史报告**：本文为 2026-04 时点的分析快照，用于回溯当时的 As-Is/To-Be 认知；当前实现以最新代码与测试为准。

（1）Loop-first 主路径：`core/harness/execution/loop.py`  
`BaseLoop.run()` 负责 session 生命周期与治理 hook 触发，`ReActLoop` 负责 reason→act→observe，并在工具调用前后触发 approval hooks。  
已具备最小观测闭环：工具失败率过高触发 PAUSED、token 高占用时 best-effort 压缩 messages。

（2）LangGraph 子系统：`core/harness/execution/langgraph/*`  
存在 GraphConfig / callbacks / ExecutionTrace（含 checkpoint 记录），可作为图执行主路径的雏形。

---

## 2) Harness 分析（执行编排/治理运行时）

### 2.1 现状优势（已对齐最佳实践的部分）
- **治理点已接线**：SESSION/CONTRACT/APPROVAL/STOP 等 HookPhase 进入主执行链路，可阻断（关键中台能力）。
- **最小观测闭环**：出现了“采集→规则→动作”雏形（虽然阈值内置，但形态对）。
- **具备图化雏形**：LangGraph 目录与 checkpoint/restore 回调机制已存在，为 Graph-first 收敛提供基础。

### 2.2 主要差距（与 2025–2026 中台最佳实践对照）
1) **Graph-first 未确立**：Loop 与 Graph 两套并存，主路径偏 Loop-first。最佳实践通常 Graph/DAG 为主，Loop 作为 node 策略。
2) **Checkpoint 不可持久化**：checkpoint 主要在 trace 内存结构；缺少 store（DB/Redis/S3）与 resume/replay 语义。
3) **事件模型不统一**：EventBus 存在，但未强制在 Loop/Graph 关键节点 emit 标准事件；审计事件多为 context 追加。
4) **策略引擎未成为“必经 gate”**：PolicyEngine 存在但与 loop/hook 的关系偏“可选组件”，缺少统一决策/审计输出。

### 2.3 建议路线（Harness）
P0（1–2 周）：把策略/审计事件贯通到主执行链路  
- 定义统一事件 schema（ToolCall、SkillCall、PolicyDecision、ApprovalDecision、LoopStep 等）
- Loop/Graph 执行强制 emit（EventBus + OTel）
- PolicyEngine 作为执行前 gate（对工具/技能/LLM 调用统一评估），并写入审计

P1（2–4 周）：checkpoint store + 可恢复执行  
- 引入 CheckpointStore（SQLite 起步即可），实现 resume/replay 最小语义
- 定义幂等策略：工具调用需要可重复/或具备 side-effect 标注与防重放机制

P2（持续演进）：Graph-first 收敛  
- 将 ReActLoop 封装为 Graph node；将多代理/计划执行等纳入图编排，统一观测/审计

## 3) Agent 分析（能力装配与执行单元）

### 3.1 现状优势
- **执行委托收敛**：主要 Agent（ReAct/PlanExecute 等）通过 `BaseAgent.execute()` → loop.run 执行，避免各自实现死循环。
- **状态模型已向 canonical 收敛**：management 与 registry 默认状态统一为 harness canonical（AgentStateEnum）。
- **多代理协作模式齐备**：coordination patterns 已实现并接入 MultiAgent。

### 3.2 主要差距
1) **Agent “控制面能力”不足**：缺少租户级隔离、配额/成本归集、版本/发布治理的明确契约。  
2) **Agent 状态与执行记录不可回放**：缺少“执行 trace + 事件源”的统一落盘模型。  
3) **ConversationalAgent 特例路径**：保留 override 合理（单轮对话），但中台最佳实践通常也会纳入统一事件/审计链路（至少要 emit 语义事件）。

### 3.3 建议路线（Agent）
- 将 Agent 执行输出标准化为 `ExecutionRun`（包含 trace_id、policy decisions、tool/skill/llm spans、cost/tokens）
- 所有 Agent（包括 Conversational）进入统一 event pipeline（OTel + 审计事件）
- 逐步引入租户/项目维度：权限、配额、审计、观测全部按维度分桶

---

## 4) Skill 分析（可复用能力单元）

### 4.1 现状优势
- **版本回滚闭环已可验收**：active_version 可查询，rollback 会影响实例 config（避免“回滚成功但行为不变”）。
- **执行器具备 timeout/record**：SkillExecutor 记录执行记录并统计到 registry。
- **inline/fork 双模式**：fork 使用 ConversationalAgent 子 agent 执行，符合“复杂能力隔离执行”的方向。

### 4.2 主要差距
1) **Skill “包/目录化”尚未落地**：文档里提到的 SKILL.md + references/scripts 的模式是行业趋势，但现在仍偏 registry+python class。
2) **Skill 合约与 schema 不够强**：validate/params 存在，但缺少统一 schema/版本兼容检查/回归评测机制。
3) **Skill 变更缺少评测回归**：中台最佳实践里，skill/agent/prompt 的发布必须绑定 eval 套件。

### 4.3 建议路线（Skill）
- 以“目录化 skill”作为中期目标（SKILL.md 元数据、handler、references、scripts），并提供 discovery + 版本管理
- 每个 skill 发布必须带：schema、兼容声明、回归用例（最小黄金集）

## 5) Tools 分析（外部能力接口）

### 5.1 现状优势
- **统一执行封装（正确）**：`BaseTool._call_with_tracking()` 做了参数校验/权限/超时/统计/可选 tracing，且 stats/registry 已加锁（线程安全最小集）。
- **权限闭环已落地**：permission_manager 注入 + 执行入口默认 seed + 授权 API。
- **工具召回（TokMem）存在**：对“工具选择”做了系统化尝试。

### 5.2 主要差距
1) **风险分级与安全策略未平台化**：现在更多是扫描/审批点，缺少“工具风险等级 → 默认策略”的体系（中台常见做法）。
2) **工具副作用与可重放语义**：若要 checkpoint/replay，工具需声明 side-effect、幂等性或防重放策略。
3) **工具市场化/安装升级**：当前 tools 更像内置模块；中台最佳实践通常走 manifest/包管理（与 MCP 强相关）。

### 5.3 建议路线（Tools）
- 为 tool schema 增加：risk_level、side_effect、idempotency、rate_limit_cost、required_permissions 等元信息
- 将 Approval/PolicyEngine 与 tool metadata 绑定（高风险默认审批/隔离执行）

## 6) MCP 分析（工具协议与生态）

### 6.1 现状优势
- **core/apps/mcp 已实现 server/client/adapter**：能把 MCP Tool 适配为本地 BaseTool，也能把本地工具暴露为 MCP Server（SSE + JSON-RPC）。
- **infra/mcp 也提供 transport/client**：具备 stdio/http/websocket 传输实现（已更完整）。
- **文档明确双栈现状与迁移口径**：避免误用（短期以 core 为主）。

### 6.2 主要差距（与 2025–2026 MCP 生态最佳实践对照）
1) **双栈重复**：core 与 infra 两套 MCP 代码长期会造成协议/鉴权/transport/测试重复维护。
2) **鉴权/隔离未落地**：文档提 OAuth/恶意包扫描，但实现侧仍偏基础协议层（中台需要强治理）。
3) **工具治理未完全统一到中台控制面**：MCP 工具进入 ToolRegistry 后，应自动获得权限、审计、风险策略、观测事件等统一能力。

### 6.3 建议路线（MCP）
P0：明确单栈（推荐以 infra.mcp 为 transport 底座，core 保留 thin adapter 与 ToolRegistry 集成）  
P1：MCP 工具进入 registry 时自动接入：权限→策略→审计→OTel  
P2：对 MCP server 引入“server trust model”（白名单/签名/隔离执行/资源访问限制）

---

## 7) 全局级问题（跨 Harness/Agent/Skill/Tools/MCP）

### 7.1 “控制面”缺口是最大差距
最佳实践 AI 中台通常有明确控制面（Policy、Tenant、Quota/Cost、Audit、Release/Evals）。  
当前工程的能力更多分散在 server seed、hook、部分 policy 规则里，需要平台化收敛。

### 7.2 评测回归缺位（中台分水岭）
建议最先建立：
- 黄金集（关键任务/工具调用链/安全用例）
- 回归流水线（每次变更跑：质量 + 成本 + 延迟 + 安全）

---

## 8) 优先级路线图（建议）

### P0（最先做，收益最大）
1) 统一事件/审计模型，并贯通到 Loop/Graph/Tool/Skill/MCP  
2) 策略引擎成为统一 gate（预算、权限、审批、风险分级）  
3) 引入“工具风险分级 + 默认策略”

### P1（中期平台化关键）
1) CheckpointStore + resume/replay（Graph-first 方向）  
2) 多租户/项目隔离：权限/审计/配额/观测按维度分桶  
3) MCP 单栈收敛与信任模型

### P2（长期差异化）
1) Skill 目录化/市场化（SKILL.md + scripts + references）  
2) 评测中台（离线 eval + 在线监控 + 回归门禁）  
3) 成本治理（token/cost attribution + 限流/熔断/降级策略）
