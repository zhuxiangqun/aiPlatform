## 2. 自动化发现的问题（按严重度）

### P0（高优先级）：大面积断链（185）
主要形态：
1) `/docs/by-role/*` 中跨层链接使用了错误的相对路径层级  
   - 例如：`docs/by-role/developer/index.md` 中 `../../aiPlat-infra/...` 实际会解析到 `docs/aiPlat-infra/...`（不存在），通常应为 `../../../aiPlat-infra/...`
2) `aiPlat-app/docs/index.md` 中跨层链接层级不正确  
   - 例如：`../aiPlat-core/docs/index.md` 实际解析到 `aiPlat-app/aiPlat-core/...`（不存在），通常应为 `../../aiPlat-core/...`
3) `docs/index.md` 内某些链接指向错误的文档根  
   - 例如 `skills/file-format.md` 解析为 `docs/skills/file-format.md`，但真实文件在 `aiPlat-core/docs/skills/file-format.md`

**影响**：读者无法通过文档导航到真实材料，且会误判模块存在性/成熟度。  
**建议修复策略**：
- 统一“相对路径基准”：所有跨层链接都从“仓库根”角度设计（推荐显式使用 `../..` 到根再进入目标层），并用脚本批量校正；
- 在 CI 加入 link-check（仅校验 repo 内相对链接）。

### P1（中优先级）：文档示例 import 与仓库内代码不匹配（18）
典型问题：
- `aiPlat_platform` / `aiplat_platform` / `aiPlat_app`：在本仓库中不存在对应包目录（与“platform/app 只有 docs”现状一致）
- `core.apps.tools.hooks...`：仓库中 `aiPlat-core/core/apps/tools` 存在，但 `hooks` 子模块不存在（文档领先或实现已迁移）
- `from harness.config import HarnessConfig`：仓库中 `aiPlat-core/core/harness` 下不存在 `config.py`（需改为实际存在的配置入口，或删掉示例）
- infra 层多处示例引用了不存在的子模块：
  - `infra.database.oracle`
  - `infra.config.sources.consul`
  - `infra.network.discovery.*`
  - `infra.mcp.config`
  - `infra.observability.exporters.*`
  - `infra.monitoring.health`
  - `infra.logging.outputs.base` / `infra.logging.formatter`

**影响**：读者照抄示例会直接报错，降低文档可信度。  
**建议修复策略**：
- 对 infra/core docs：将“示例代码”改为基于真实存在模块的最小可运行例子（或标注为伪代码）。
- 对 platform/app docs：若短期不引入代码，删除导入示例，改成“接口契约 + 调用链”。

## 3. 抽样核对的关键断言（Spot-check）

### Runtime / Gateway / Deployment 的归属口径
已在前序修复中统一为：
- Core Runtime（执行引擎）= `aiPlat-core (Layer 1)`
- Message Gateway（渠道接入）= `aiPlat-app (Layer 3)`
- API Gateway（认证/路由/治理）= `aiPlat-platform (Layer 2)`
- Deployment：platform 仅可描述“控制面”，执行面在 infra/ops

> 当前仓库依旧存在的主要风险：platform/app 没有代码实现，因此这些口径更多是“架构定义正确”，不是“实现已验证”。

## 4. 下一步建议（按投入产出排序）

### 建议 A：先把 P0 断链清零（推荐）
1) 先修 `docs/`（系统级文档）里的跨层链接
2) 再修 `aiPlat-app/docs` 和 `aiPlat-platform/docs` 的跨层链接
3) 通过脚本/CI 固化，避免回归

### 建议 B：把所有“示例代码”分级
- **可运行示例**：必须能在仓库内 import 成功（否则不准写成可运行）
- **伪代码示例**：必须明确标注 “伪代码/示意”，并避免出现真实 import 路径

### 建议 C：明确 platform/app 的仓库现状
二选一：
1) 把 platform/app 实现代码补齐纳入仓库；或
2) 在对应 docs index 顶部加显式声明：`本层当前仅包含设计与接口契约，代码实现未纳入本仓库/尚未落地`
