### 2.2 Agent 执行路径（简化版）
`POST /agents/{id}/execute` 典型路径：
1) AgentRegistry 获取实例  
2) PermissionManager 检查 EXECUTE  
3) 注入 model / approval_manager / tools / skills  
4) `agent.execute(context)` → `BaseAgent.execute()` → `loop.run()`  
5) loop 内 `reason → act → observe`，工具调用前后触发 hooks  
6) server 层记录 execution/history（当前为内存 dict）

### 2.3 Skill 执行路径（简化版）
`POST /skills/{id}/execute` 典型路径：
1) PermissionManager 检查 EXECUTE  
2) SkillManager 创建/更新执行记录  
3) SkillExecutor 执行（inline 或 fork）  
4) 回写执行状态并返回

## 3. 分模块评估（Harness / Agent / Skill / Tools / MCP）

> 评价维度：是否符合 2025–2026 中台最佳实践：**可治理、可观测、可回归、可扩展、可一致、可隔离**。

### 3.1 Harness（执行编排/治理运行时）

**现状优势**
- HookPhase 已进入主执行链路（SESSION/CONTRACT/APPROVAL/STOP），具备阻断语义（治理闭环已接线）
- 最小观测控制闭环已存在（工具失败率→PAUSED、token 高占用→压缩 messages）
- LangGraph 子系统存在（checkpoint/restore 回调），具备 Graph-first 演进基础

**主要差距**
- **Graph-first 未确立**：Loop 与 Graph 两套形态并存，主路径偏 Loop-first
- **Checkpoint 不可持久化**：更多是 trace 内存记录，缺少 store + resume/replay 语义
- **事件模型未统一落盘**：EventBus 存在但未强制接入 Loop/Graph 关键节点；审计多写入 context

**关键问题（来自你的分析，证据已确认）**
- Action 解析脆弱（`loop.py _act()` 依赖 `"ACTION:"` + `":"` 分割）

**建议（Harness）**
P0：将 tool/skill/llm/policy/approval 统一事件化（标准事件 schema + OTel/OpenInference 风格）  
P1：CheckpointStore（SQLite/Redis）+ resume/replay（含副作用声明/防重放）  
P2：Graph-first 收敛（Loop 作为 graph node）

---

### 3.2 Agent（能力装配与执行单元）

**现状优势**
- 主要 Agent 委托 Harness Loop 执行（避免各自实现死循环）
- canonical 状态模型已收敛到 `AgentStateEnum`（management/registry 对齐）
- 多代理协作 patterns 已实现且可用

**主要差距**
- Agent 执行记录/审计仍偏“server 内存态”，难以满足多 worker 与回放
- ConversationalAgent 特例路径合理，但需确保进入统一观测/审计链（至少 emit 事件）

**你指出的“双重委托”**
- ReActAgent.execute() 先注入 tools 再调用 super().execute()（BaseAgent.execute() 又会注入 model/skills/tools）
- 这是**可重构的代码味道**，更偏 P2（可读性与一致性优化），不是 P0 生产事故点

---

### 3.3 Skill（可复用能力单元）

**现状优势**
- 版本回滚语义闭环可验收（active_version 可查询，回滚影响后续执行）
- SkillExecutor 支持 timeout + execution record
- inline/fork 两种模式存在（fork 通过 ConversationalAgent 执行，符合隔离思路）

**主要差距**
- 仍偏“registry + python class”形态；行业更趋向“目录化 skill（SKILL.md/handler/scripts/references）”
- 缺少“强 schema + 兼容策略 + 回归 eval”发布门禁（中台分水岭）

---

### 3.4 Tools（外部能力接口）

**现状优势**
- BaseTool 统一 wrapper（校验/权限/超时/统计/trace），且 stats/registry 已做最小线程安全
- 权限闭环已形成（默认 seed + 授权 API + 执行前检查）

**主要差距**
- 缺少 tool 风险分级体系（risk_level/side_effect/idempotency/rate_limit_cost 等元信息）
- 若要 checkpoint/replay，需要工具声明副作用/幂等性与防重放策略

**你指出的 skill 匹配误触发**（本质是“路由契约不足”）
- 当前 skill 触发采用 substring（`if skill_name in reasoning.lower()`），存在误触发风险  
建议升级为：结构化选择（skill_name 显式字段）或策略路由（select_tool/select_skill）节点

---

### 3.5 MCP（工具协议与生态接入）

**现状优势**
- core/apps/mcp 已实现 server/client/adapter，可把 MCP tool 纳入 ToolRegistry
- infra/docs 明确存在“core/infra 双栈”，并给出短期口径

**主要差距**
- 双栈重复长期会带来演进成本（transport、鉴权、测试、治理重复）
- 鉴权/信任模型/隔离未形成：文档有方向，但实现仍偏协议/传输层
- MCP 工具进入 registry 后，应自动接入：权限→策略→审计→OTel（目前还不够“统一管线化”）

## 4. 关键问题清单（最终版：融合你的清单 + 口径校正）

| 问题 | 严重性 | 影响 | 最终口径 |
|---|---|---|---|
| server.py 全局执行历史 dict（非线程/进程安全） | 高（P0） | 并发/多 worker 数据丢失、重启丢失、内存泄漏 | ✅确认，必须持久化替代 |
| Action 解析依赖文本格式（ACTION: tool: args） | 中（P0/P1） | LLM 输出变化导致无法调用工具 | ✅确认，建议升级为结构化 tool calling |
| Skill 触发 substring 匹配 | 中（P1） | 误触发导致错误能力执行 | ✅确认，建议改为显式路由/结构化输出 |
| MultiAgent pattern + fallback 双路径 | 低→中（P1） | 语义漂移/维护成本 | ✅确认，建议统一执行路径 |
| HookManager 默认 print / 异常处理粗糙 | 低（P2） | 观测与排障困难 | ✅确认，建议 logging+事件化 |
| ConvergenceDetector “未接入” | 中（口径需校正） | 文档/实现不一致 | ⚠️更准确是：Graph 路径已用，MultiAgent 类执行路径未统一，属于“路径不一致” |
| Agent 执行路径双重委托 | 低（P2） | 可读性/一致性差 | ✅属于可重构项，不是首要生产风险 |

## 5. 是否“最佳实践 AI 中台”？最终判断

**当前不是最佳实践终态**，但它具备中台化的正确骨架。  
要达到 2025–2026 常见的“生产级 AI 中台”标准，关键不是增加更多 agent/skill，而是把下面三件事平台化：

1) **结构化契约**：工具/技能选择与调用必须结构化（否则不可治理、不可回归）。  
2) **可观测与审计事件统一管线**：把每次运行变成可检索的 run/trace/event（并对接 OTel）。  
3) **编排收敛与可恢复执行**：Graph-first、checkpoint store、resume/replay（含副作用治理）。

## 6. 改进路线图（带验收标准）

### P0（必须优先，面向生产风险）
1) **执行记录持久化替换 server 全局 dict**
   - 方案：SQLite/Redis/DB 任一，至少做到：按 execution_id 查询、按 agent_id 分页 history
   - 验收：并发压测下历史不丢；重启后仍可查询

2) **工具调用改为结构化（Tool Calling / JSON Mode）**
   - 方案：模型输出强制 JSON schema（tool_name + args），解析失败走纠错/重试，不直接 silent fail
   - 验收：构造 5 种非标准输出仍可恢复到结构化调用；具备回归测试集

### P1（平台化收敛）
3) **MultiAgent 统一执行路径**
   - 方案：统一走 coordination patterns（或 graph），将 fallback 作为显式降级策略并事件化
   - 验收：同一输入在 pattern/fallback 不再产生不同语义；测试覆盖

4) **统一事件/审计模型，并接入 OTel**
   - 方案：定义标准事件 schema（policy/approval/tool/skill/llm/retrieval），EventBus 落盘/导出
   - 验收：每次执行可生成可查询的 run（trace_id/decision/event 列表）

### P2（长期最佳实践目标）
5) **Graph-first + CheckpointStore + Resume/Replay**
   - 验收：支持暂停后恢复、失败后重放；工具副作用声明生效

6) **Skill 目录化与发布门禁（Eval/Regression）**
   - 验收：每个 skill 版本发布必须带 schema + 最小黄金集回归
