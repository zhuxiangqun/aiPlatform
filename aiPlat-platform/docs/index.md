# aiPlat-platform 文档索引

## 概述

aiPlat-platform 是 AI 中台的平台服务层，负责提供面向业务的平台级服务。本层在核心层之上构建，提供 API 网关、认证授权、租户管理、消息队列等平台能力，是连接核心层和应用层的桥梁。

## 架构定位

### 层级关系

aiPlat-platform 位于四层架构的第三层（Layer 2），承上启下：

- **向上依赖**：为 aiPlat-app 提供平台服务
- **向下依赖**：依赖 aiPlat-core 提供的核心 AI 能力
- **被依赖方**：aiPlat-app

### 职责边界

本层负责：

- API 网关与路由管理
- 用户认证与授权
- 租户管理与资源隔离
- 消息队列与事件驱动
- 服务注册与发现
- 计费与配额管理
- 治理与审计

本层不负责：

- 智能体编排与执行（由 aiPlat-core 提供）
- 技能与记忆管理（由 aiPlat-core 提供）
- 数据库连接池管理（由 aiPlat-infra 提供）
- 前端界面与用户交互（由 aiPlat-app 提供）

## 核心模块

### api - API 服务

API 服务模块提供 RESTful API 和 GraphQL API 的定义、实现和管理。

**核心概念**：

- **Endpoint**：API 端点，定义 API 的访问路径和方法
- **Route**：路由，定义请求到处理器的映射
- **Handler**：处理器，处理具体的 API 请求
- **Middleware**：中间件，处理请求前后的通用逻辑

**API 类型**：

- **智能体 API**：智能体创建、运行、管理
- **技能 API**：技能注册、查询、执行
- **知识库 API**：知识库创建、更新、检索
- **工具 API**：工具注册、查询、调用
- **会话 API**：会话创建、管理、历史查询
- **管理 API**：配置管理、监控查询、日志查询

**主要能力**：

- API 定义与实现
- 请求路由与分发
- 请求验证与转换
- 响应格式化与错误处理
- API 版本管理
- API 文档生成

**设计原则**：

- API 应该遵循 RESTful 设计规范
- API 应该有清晰的版本管理
- API 应该有完整的错误处理
- API 应该有详细的文档说明

**相关文档**：[api 模块文档](./api/index.md)

### auth - 认证授权

认证授权模块负责用户身份认证和权限管理，保障系统安全。

**核心概念**：

- **User**：用户，系统的使用者
- **Role**：角色，权限的集合
- **Permission**：权限，对资源的操作许可
- **Token**：令牌，认证凭证
- **Session**：会话，用户登录状态

**认证方式**：

- **密码认证**：用户名密码登录
- **OAuth2**：第三方登录（GitHub、Google 等）
- **API Key**：API 密钥认证
- **JWT**：JSON Web Token 认证
- **SAML**：企业单点登录

**授权模型**：

- **RBAC**：基于角色的访问控制
- **ABAC**：基于属性的访问控制
- **ACL**：访问控制列表

**主要能力**：

- 用户注册与登录
- 密码管理与重置
- 令牌生成与验证
- 权限检查与校验
- 会话管理与超时
- 审计日志记录

**设计原则**：

- 认证与授权应该分离
- 密码应该加密存储
- 令牌应该有过期时间
- 敏感操作应该有二次验证

**相关文档**：[auth 模块文档](./auth/index.md)

### tenants - 租户管理

租户管理模块负责多租户系统的租户隔离和资源管理。

**核心概念**：

- **Tenant**：租户，独立的业务实体
- **Namespace**：命名空间，资源隔离边界
- **Quota**：配额，资源使用限制
- **Resource**：资源，租户可使用的计算资源

**隔离级别**：

- **逻辑隔离**：通过命名空间和标签实现
- **数据隔离**：通过数据库 schema 或表前缀实现
- **资源隔离**：通过资源配额和限制实现

**主要能力**：

- 租户创建与配置
- 租户资源分配
- 租户数据隔离
- 租户配额管理
- 租户使用统计
- 租户迁移与删除

**设计原则**：

- 租户之间应该完全隔离
- 租户配置应该支持自定义
- 租户资源应该有配额限制
- 租户操作应该有审计日志

**相关文档**：[tenants 模块文档](./tenants/index.md)

### gateway - 服务网关（API Gateway）

服务网关模块负责 **API Gateway（Layer 2）** 的请求路由、认证接入、限流、熔断等网关能力。

> **说明**：多渠道接入的“消息网关（Message Gateway）”位于 **aiPlat-app（Layer 3）**；本处的 gateway 指的是 API 网关。

**核心概念**：

- **Router**：路由器，定义请求到服务的映射
- **Filter**：过滤器，处理请求前后的逻辑
- **RateLimiter**：限流器，控制请求速率
- **CircuitBreaker**：熔断器，防止级联故障
- **LoadBalancer**：负载均衡器，分发请求

**主要能力**：

- 请求路由与转发
- 请求限流与熔断
- 请求重试与超时
- 负载均衡与故障转移
- 请求日志与追踪
- SSL 终止与证书管理

**设计原则**：

- 网关应该是无状态的
- 网关应该支持水平扩展
- 网关应该有完整的监控
- 网关应该支持配置热更新

**相关文档**：[gateway 模块文档](./gateway/index.md)

### messaging - 消息服务

消息服务模块负责消息队列和事件驱动架构的支持。

**核心概念**：

- **Message**：消息，传递的数据单元
- **Queue**：队列，消息的存储容器
- **Topic**：主题，消息的分类
- **Producer**：生产者，发送消息
- **Consumer**：消费者，接收消息
- **Subscription**：订阅，消费者对主题的关注

**消息模式**：

- **点对点**：消息发送到队列，单个消费者处理
- **发布订阅**：消息发送到主题，多个消费者订阅
- **请求响应**：消息发送后等待响应

**主要能力**：

- 消息发送与接收
- 消息持久化与重试
- 消息顺序保证
- 消息去重与幂等
- 死信队列处理
- 消息追踪与监控

**设计原则**：

- 消息应该是不可变的
- 消息应该有明确的格式定义
- 消息处理应该支持幂等
- 消息应该有重试机制

**相关文档**：[messaging 模块文档](./messaging/index.md)

### deployment - 部署服务（控制面）

部署服务模块在平台层的定位是**部署控制面 API**：负责权限/策略校验、任务下发与状态聚合。

> **重要说明**：部署的“执行面”（K8s/Helm/资源操作/镜像构建等）属于 **aiPlat-infra / 运维体系**，不应落在 platform。

**核心概念**：

- **Deployment**：部署，服务的发布过程
- **Version**：版本，服务的版本号
- **Environment**：环境，部署的目标环境
- **Strategy**：策略，部署的方式

**部署策略**：

- **滚动更新**：逐步替换旧版本
- **蓝绿部署**：新旧版本并存，流量切换
- **金丝雀发布**：小流量测试新版本
- **A/B 测试**：不同版本对比测试

**主要能力（控制面）**：

- 部署任务创建/回滚/查询
- 部署策略参数校验（滚动/蓝绿/金丝雀）
- 权限、租户隔离、审计
- 聚合来自执行器/基础设施的状态（只读视图）

**设计原则**：

- 部署应该是可重复的
- 部署应该支持回滚
- 部署应该有完整的日志
- 部署应该支持灰度发布

**相关文档**：[deployment 模块文档](./deployment/index.md)

### registry - 服务注册

服务注册模块负责服务的注册、发现和健康检查。

**核心概念**：

- **Service**：服务，提供特定功能的组件
- **Instance**：实例，服务的运行实例
- **Registry**：注册中心，管理服务实例信息
- **Discovery**：服务发现，查找可用服务实例

**主要能力**：

- 服务注册与注销
- 服务发现与负载均衡
- 服务健康检查
- 服务元数据管理
- 服务依赖管理
- 服务拓扑可视化

**设计原则**：

- 服务注册应该是自动的
- 服务发现应该是高效的
- 健康检查应该是可靠的
- 服务信息应该有缓存

**相关文档**：[registry 模块文档](./registry/index.md)

### billing - 计费服务

计费服务模块负责资源使用统计和费用计算。

**核心概念**：

- **Plan**：套餐，定义资源配额和价格
- **Usage**：用量，资源使用量统计
- **Bill**：账单，费用计算结果
- **Invoice**：发票，账单的正式凭证

**计费模式**：

- **按量计费**：根据实际使用量计费
- **包年包月**：固定费用，固定配额
- **阶梯计费**：用量越大，单价越低
- **预留实例**：预付费，享受折扣

**主要能力**：

- 用量统计与上报
- 费用计算与账单生成
- 套餐管理与升级
- 优惠券与折扣
- 充值与退款
- 发票开具与管理

**设计原则**：

- 用量统计应该是准确的
- 费用计算应该是透明的
- 账单应该是可查询的
- 计费应该支持多种支付方式

**相关文档**：[billing 模块文档](./billing/index.md)

### governance - 治理服务

治理服务模块负责系统的治理、审计和合规。

**核心概念**：

- **Policy**：策略，定义治理规则
- **Rule**：规则，具体的治理规则
- **Audit**：审计，操作记录和审查
- **Compliance**：合规，符合法规和标准

**治理领域**：

- **数据治理**：数据质量、数据安全、数据生命周期
- **模型治理**：模型版本、模型效果、模型合规
- **API 治理**：API 规范、API 安全、API 生命周期
- **资源治理**：资源配额、资源标签、资源清理

**主要能力**：

- 策略定义与执行
- 操作审计与追踪
- 合规检查与报告
- 风险识别与预警
- 数据脱敏与加密
- 敏感信息检测

**设计原则**：

- 治理应该是自动化的
- 审计应该是完整的
- 合规应该是持续的
- 风险应该是可控的

**相关文档**：[governance 模块文档](./governance/index.md)

## 设计原则

### 依赖原则

- **单向依赖**：只依赖 aiPlat-core，不依赖 aiPlat-app
- **接口隔离**：通过接口与核心层交互，不直接访问内部实现
- **依赖注入**：使用依赖注入管理依赖关系

### 设计原则

- **服务化**：以服务为中心，提供稳定的 API
- **可扩展**：支持水平扩展，应对流量增长
- **高可用**：支持故障转移，保证服务稳定
- **可观测**：支持监控、日志、追踪

### 编码规范

- 使用类型注解，提高代码可读性
- 使用异步编程，提高并发性能
- 使用配置化设计，支持灵活定制
- 使用日志记录，支持问题排查
- 使用单元测试，保证代码质量
