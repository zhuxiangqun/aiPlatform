## 🏗️ 架构设计

### 分层原则

基础设施层采用三层架构：

**接口层**：定义抽象接口，如数据库接口、LLM 客户端接口、向量存储接口等

**实现层**：提供具体实现，如 PostgreSQL 数据库实现、OpenAI 客户端实现、FAISS 向量存储实现等

**工厂层**：提供工厂函数，根据配置创建合适的实例

---

### 依赖管理

#### 依赖方向

**严格规则**：
- 基础设施层不依赖任何内部包
- 可以被所有上层导入（core, platform, app）
- 所有模块都可以依赖 config 模块

#### 模块依赖规则

**允许的依赖**：
- 所有模块都可以依赖 `config` 模块
- `llm` 可以依赖 `cache`（用于响应缓存）
- `vector` 可以依赖 `cache`（用于查询缓存）
- `database` 可以依赖 `cache`（用于查询缓存）
- `compute`、`memory`、`network` 可以依赖 `cache`（用于状态缓存）

**禁止的依赖**：
- 不允许循环依赖
- 不允许 `config` 依赖其他业务模块
- 不允许跨层级依赖（如 `llm` 依赖 `database`）
- 模块之间不允许非必要的直接依赖

### 接口设计

#### 数据库接口

数据库接口定义了统一的数据访问方法：
- 连接和断开连接
- 执行查询和批量操作
- 事务管理
- 健康检查

接口设计遵循最小知识原则，只暴露必要的方法。

---

#### LLM 客户端接口

LLM 客户端接口定义了统一的 LLM 访问方法：
- 对话和流式对话
- 文本嵌入
- Token 计数

接口支持多种 LLM 实现，如 OpenAI、Anthropic、DeepSeek 等。

---

#### 算力管理接口

算力管理接口定义了统一的资源管理方法：
- CPU/GPU 资源分配与释放
- 任务调度与队列管理
- 弹性扩缩容
- 资源配额管理

**接口方法**：
- `allocate(request)` - 分配算力资源
- `release(allocation_id)` - 释放资源
- `submit_task(task)` - 提交计算任务
- `list_nodes(filters)` - 列出计算节点

---

#### 内存管理接口

内存管理接口定义了统一的内存管理方法：
- 多级缓存（L1/L2/L3）
- 内存池管理
- 显存（VRAM）管理
- OOM 防护

**接口方法**：
- `allocate(request)` - 分配内存
- `release(allocation_id)` - 释放内存
- `get_stats(node_id)` - 获取内存统计
- `enable_oom_protection(threshold)` - 启用 OOM 保护

---

#### MCP 协议接口

MCP 协议接口定义了统一的工具调用方法：
- MCP 协议解析（JSON-RPC）
- MCP 客户端连接管理
- 工具发现与调用

**接口方法**：
- `connect(server_url)` - 连接到 MCP Server
- `list_tools()` - 列出可用工具
- `call_tool(name, arguments)` - 调用工具

---

#### 接口设计示例

**好的接口设计**（小而专）：
```python
class DatabaseClient(ABC):
    @abstractmethod
    async def execute(self, query: str, params: dict) -> List[dict]:
        """执行查询，返回结果"""
        pass
    
    @abstractmethod
    async def transaction(self) -> AsyncContextManager:
        """获取事务上下文"""
        pass
```

**不好的接口设计**（大而全）：
```python
class DatabaseClient(ABC):
    @abstractmethod
    async def execute(self, query: str, params: dict) -> List[dict]:
        pass
    
    @abstractmethod
    async def backup(self) -> bool:  # 不应该在这里
        pass
    
    @abstractmethod
    async def migrate(self) -> bool:  # 不应该在这里
        pass
```

### 扩展性设计

#### 支持多种实现

通过接口与实现分离，支持多种数据库、LLM、向量存储实现：
- 数据库支持 PostgreSQL、MySQL、MongoDB 等
- LLM 支持 OpenAI、Anthropic、DeepSeek 等
- 向量存储支持 FAISS、Milvus、Pinecone 等

添加新实现只需实现接口并注册到工厂函数。

---

#### 配置驱动切换

通过配置文件切换实现，无需修改代码：
- 修改配置文件中的类型字段
- 工厂函数根据类型返回对应实现
- 支持运行时动态切换

---

### 性能优化

#### 连接池设计

数据库使用连接池管理连接：
- 减少连接创建开销
- 复用连接提高性能
- 控制并发连接数
- 自动管理连接生命周期

连接池配置包括最小连接数、最大连接数、连接超时等参数。

## 📊 设计决策

### 为什么使用接口？

**问题**：如何支持多种数据库、LLM、向量存储？

**解决方案**：接口与实现分离

**优势**：
- 易于切换实现
- 易于测试（Mock）
- 易于扩展新实现
- 符合依赖倒置原则

---

### 为什么使用工厂模式？

**问题**：如何根据配置创建不同实例？

**解决方案**：工厂函数 + 配置驱动

**优势**：
- 配置驱动创建实例
- 隐藏创建复杂度
- 易于管理实例生命周期
- 支持依赖注入

---

### 为什么使用连接池？

**问题**：如何管理数据库连接以提高性能？

**解决方案**：连接池设计

**优势**：
- 减少连接创建开销
- 复用连接提高性能
- 控制并发连接数
- 自动管理连接生命周期

---

### 为什么模块之间不能循环依赖？

**问题**：模块之间如果循环依赖，会导致：
- 无法独立测试
- 难以替换实现
- 启动顺序复杂
- 难以理解代码结构

**解决方案**：所有模块只依赖 config，不相互依赖

**优势**：
- 模块可独立测试
- 易于替换实现
- 启动顺序无关
- 代码结构清晰

---

### 为什么需要显存管理？

**问题**：LLM 推理占用大量显存，需要统一管理

**解决方案**：memory 模块提供显存池，统一分配回收

**优势**：
- 避免显存碎片
- 支持多模型缓存
- 动态调整显存分配
- OOM 保护

---

### 为什么需要 MCP 协议支持？

**问题**：AI Agent 需要调用外部工具，标准化程度低

**解决方案**：mcp 模块提供统一的 MCP 协议实现

**优势**：
- 标准化工具调用
- 支持多种 MCP Server
- 统一的连接管理
- 工具发现和调用解耦

## 🔍 最佳实践

### 1. 接口定义要清晰

接口方法应该有明确的参数和返回类型，方法名要能清晰表达其功能。

---

### 2. 配置要类型安全

配置使用类型注解，确保配置值的类型正确，避免运行时错误。

---

### 3. 工厂函数要有默认值

工厂函数应该提供默认配置，简化使用，同时支持自定义配置。

---

### 4. 模块依赖要明确

新增模块时，明确其依赖关系，避免引入循环依赖。
